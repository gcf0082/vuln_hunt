# 忽略的漏洞场景

以下类别命中时直接跳过：

- **反序列化**：忽略所有 JSON 相关的反序列化风险（Jackson/Fastjson 等）。其他类型的反序列化，除非涉及高危反序列化函数（如 `ObjectInputStream.readObject`、`pickle.loads` 等），否则也忽略
- **日志注入**：影响太小，不关心
- **路径类环境变量**：环境变量来源为部署配置，攻击者无法控制。无直接证据时均视为安全。
- **敏感信息认定**：只有**明文或加密的认证凭据类信息**（密码、Token、密钥、证书私钥、Cookie等）、**电话号码**、**邮箱地址**视为敏感信息。除此以外的信息（如用户名、昵称、ID、地址、非私钥的证书字段等）不认定为敏感信息，不得因此报告漏洞。**但当数据内容无法确定时，按敏感信息处理**——即数据的具体内容在当前代码中不可见或无法追溯（如请求 body 结构未定义、data/msg 等通用容器的实际内容不可追踪），因无法排除其中包含敏感信息的可能。
- **CRLF 注入**：`header.put(key, value)` 等 HTTP header 设置操作，底层框架已统一处理 CRLF 字符过滤，不视为漏洞
- **查询参数注入**：`params.put(key, value)` 等查询参数设置操作，底层框架已统一处理，不视为漏洞
- **资源标识作存在性判断或查询后返回**：用户提供的资源标识（如 ID、编号）用来做存在性判断或查询，查询不到就返回，不会走到注入类操作路径的，不认为是漏洞。
- **局部模块缺少认证鉴权**：分析的只是一个模块或子系统而非完整独立系统，接口缺少认证鉴权不一定是漏洞——认证鉴权可能在网关、过滤器、拦截器、框架层统一处理。除非分析对象是完整的独立系统（含 Web 服务启动、完整认证流程），否则认证鉴权缺失一般不认为是漏洞。
