# source-analyze-vulnerability

## 定位

- **用户负责**：给一个具体的攻击面产物文件（位于 `.vuln_agent_output/analyzed_surfaces/`），或描述想分析的目标（直接分析模式）
- **skill 负责**：读文件，按 `references/constraints.md` 的范围约束做深入分析；按 `references/vuln_rules_index.md` 的哨兵索引**按需**加载具体规则文件；按本文「输出格式」一节的模板落盘 VULN / DISMISSED / CLEAN / SUSPECTED 四类结论

skill 根据用户任务做漏洞分析，输入是 `analyzed_surfaces/` 下的一个或多个文件，按规模和复杂度智能分配处理，输出对应 `vuln_findings/` 下的结论文件。

## 路径约定

本 skill 涉及的所有目录**统一在 `.vuln_agent_output/` 下**，当前工作目录视为被分析项目根：

```
.vuln_agent_output/
├── analyzed_surfaces/                    ← 输入（默认模式的源文件）
├── planned_vuln_tasks/                   ← 输入（source-plan-vuln-tasks 产物，默认模式时按需生成）
├── vuln_findings/                        ← 输出（产物落盘位置）
├── meta/
│   ├── batches/{surface_stem}/           ← 复杂攻击面子任务中间产物
│   └── error/source-analyze-vulnerability.md    ← 失败日志
└── temp/
    └── scripts/                          ← 临时脚本（用完即弃）
```

## 任务澄清

### 何时反问

用户显式调用 `source-analyze-vulnerability` 但**未指定 `surface_file`** 时，**必须**先反问：

> 分析 `analyzed_surfaces/` 下的所有文件，还是指定文件？

- 用户给了文件 → 直接进入工作流程
- 用户说"全部" → 按规模和复杂度智能分配处理所有 surface

### 直接分析模式（非 analyzed_surfaces/ 输入）

两种情况下切换到**直接分析模式**：

1. **无 surface 文件**：用户在消息里直接描述分析目标（含接口/入口信息、关注漏洞模式等）
2. **文件不在 analyzed_surfaces/ 下**：用户给了文件路径，但不在 `analyzed_surfaces/` 中（如直接给源码文件、配置文件等）

**不要预设**用户会怎么输入：可能是路径、可能是标识符、可能是代码片段、可能是模糊的业务概念，也可能是其它任何形式——**按用户实际给出的内容灵活处理**。

**自动分配文件名**：直接分析模式下产物文件名由 skill 根据输入派生（来自文件路径、内容特征或用户描述），不要求与任何 `analyzed_surfaces/` 文件同名。

### 已分析文件优化

如果用户指向 `vuln_findings/` 下的**已存在**结论文件，或用户措辞表达"优化/改进/补充/重写/更新/..."等意图，切换到**优化模式**——基于原文件 + 相关源码 + 用户新增要求做修改，**原地覆盖**同名文件。

**不要预设**优化范围——用户可能要求补充、新增、删除、改写、重新结构化，**按用户实际要求灵活处理**。

**默认约束**：优化应**基于原文件已覆盖的分析范围**进行——补充、修正、细化都围绕原文件所针对的攻击面/漏洞模式展开，**不**应插入与原分析无关的新内容。

**遇到明显无关的插入请求时**：skill **必须**先反问用户是否继续，得到明确确认后再执行。模糊 / 边缘相关的情况由 skill 自己判断。

## 规划与跳过

**默认行为**：当输入是 `analyzed_surfaces/{surface_file}`（默认模式）时，默认**先调用 `source-plan-vuln-tasks` 生成 `planned_vuln_tasks/`，再以每个 plan 任务为分析子任务展开详细分析**。

### 何时跳过 source-plan-vuln-tasks

满足以下**任一**条件即跳过规划，直接按用户任务本身展开分析：

- 用户指定了具体漏洞类型（命令注入 / SQL 注入 / 路径穿越 / XSS / SSRF / XXE / 反序列化 / 弱加密 / 越权 / ...）
- 用户指定了具体参数名（filename / cmd / id / url / path / ...）
- 用户问了具体的 yes/no 问题（"是否存在 X"、"是否能 X"、"会不会 X"）
- 用户引用了具体函数 / 位置（"XController 的 foo 方法"、"UserService.java:48"）

### 自动跳过的模式

下列模式**永远跳过** source-plan-vuln-tasks：

- **直接分析模式**：用户给了代码 / 描述但没给文件 → 用户已经提供了具体输入
- **优化模式**：改 `vuln_findings/` 下的已存在文件 → 不重新规划

### 派发方式

- `planned_vuln_tasks/` 下存在 `{surface_stem}-{n}.md` → **直接读**产物，跳过派发
- 仅 `_no_tasks-{surface_stem}.md` 存在 → **跳过本次分析**，该 surface 无需分析
- 无任何文件 → 派发 `source-plan-vuln-tasks` subagent（`subagent_type=source-task-planner`）→ 等返回：
  - 产出了 `{surface_stem}-{n}.md` → 每个 = 1 个子任务
  - 仅产出了 `_no_tasks-{surface_stem}.md` → **跳过本次分析**

子任务集合确定后，每个任务独立走后续「分批分析 → 举证核查 → Payload」流程。

## 预加载

启动时**必须**先按顺序读取：

1. `references/constraints.md`（项目分析约束：非分析目标、临时脚本位置）
2. `references/vuln_rules_index.md`（漏洞规则索引：哨兵模式，**仅作索引**）

`references/vuln_rules/*.md` **不在启动预加载范围**——这些是**按需加载的具体规则**：

- 启动时**禁止**一次性加载该目录下所有规则文件
- 启动时**禁止**加载与当前任务无关的规则文件
- **仅当**任务疑点与索引中某模式匹配时，**才加载**对应规则文件深入分析
- **禁止**浏览索引寻找其他分析方向——以任务为重，疑点所指即分析目标

`references/non-vuln-scenarios.md` **不在启动预加载范围**——它是**已知非漏洞场景参考**，在判定为 CLEAN / DISMISSED 时查阅，避免把非漏洞场景误判为漏洞。

未读完 `constraints.md` 和 `vuln_rules_index.md` 不允许动笔。

## 工作流程

1. **预加载**：按上述顺序读 `references/constraints.md` 和 `references/vuln_rules_index.md`
2. **解析任务**：从用户调用消息中提取：
   - `surface_file`（如已指定）
   - `task_content`（本次分析任务——漏洞模式 / 规则 / 关注点 / 范围等）
3. **读攻击面文件**：打开 `.vuln_agent_output/analyzed_surfaces/{surface_file}` 拿到入口/接口的基本信息（URL、来源、参数、分类、调用链、关键控制点等）
4. **规划与拆分**：
   - 评估用户任务是否具体（参考「规划与跳过」节）
   - 满足跳过条件 / 直接分析 / 优化模式 → 用户任务或原文件 = 1 个子任务
    - 否则 → 检查 `planned_vuln_tasks/` 下该 surface 的文件：
      - 存在 `{surface_stem}-{n}.md` → 读所有任务，每个 = 1 个子任务
      - 仅 `_no_tasks-{surface_stem}.md` → **跳过本次分析**
      - 无文件 → 派发 `source-task-planner` subagent → 等返回，按同样规则判断
5. **列规则与疑点**（针对每个子任务）：根据 `vuln_rules_index.md` 列出与当前任务匹配的规则模式；从攻击面文件初步识别的高危漏洞优先规划为独立子任务
6. **分批分析**：按优先级分批，每批不超过 3 个模式：
   - 入口 → 参数流向
   - 中间处理逻辑
   - 敏感操作槽
   - 每段分析产物写入 `.vuln_agent_output/meta/batches/{surface_stem}/`（按文件名隔离避免并行冲突）
7. **举证核查**：检查每条结论是否有代码级事实支撑
8. **Payload 迭代验证**：仅对"确认"漏洞生成 payload 并验证可行性
9. **收集所有批次结果，写入最终输出目录**

## 分析方法

### 基本规则

**核心原则：基于事实，不基于假设。** 每一项认定的背后必须有代码级的事实支撑。有校验不代表安全——必须分析校验是否存在切实可行的绕过方式。

> **分析范围说明**：当前分析仅覆盖系统的局部模块，**从数据库、消息队列等存储读取的数据**可能由其他模块或接口的外部用户写入。
>
> **URL 路径段拼接**：参数来源于 URL 路径段（path 部分，非 query/body/header）又拼接到其他 URL 的，系统框架已统一处理，不视为漏洞场景。

- 每条结论必须有代码级事实支撑
- 信息不足必须用 SUSPECTED，禁止降级为 CLEAN。CLEAN 仅限本身无风险操作的条目
- 注入类问题 **从数据库获取的输入**，无明显证据证明安全的，按 SUSPECTED 处理
- 追踪入参到敏感操作槽的完整调用链
- **全面分析**：必须逐一检查每个请求参数的流向和每个关键控制点，不能因发现某个参数存在漏洞就停止对其他参数的分析
- **全链路数据流优先**：漏洞分析要**主要关注整个数据流和调用链**（入口 → 中间处理 → 敏感操作槽），**不能只看局部**——一处看似无害的拼接，在整条调用链的某个环节可能成为漏洞。**重点排查**参数如何在层与层之间传递、中间件/校验函数是否被绕过、敏感操作之前的最后一道防线是否真的有效。
- **注入类漏洞必出文件**：只要输入来自外部可控，不论是否有漏洞都必须生成独立的结论文件（VULN / DISMISSED / CLEAN / SUSPECTED），禁止静默跳过
- **贴出关键代码举证**：每条结论必须贴出**关键代码片段**（`{文件路径:行号}` + 原文）作为事实依据，不得仅靠自然语言描述。**尤其当认为校验存在被绕过的问题时，必须贴出校验代码本身**（含注解式校验如 `@NotNull`、`@Size`、`@Pattern`、`@Valid`），否则不构成有效证据。
- **C/C++ 项目内存安全**：如果分析对象是 C/C++ 项目，还需关注缓冲区溢出、越界访问、释放后使用、空指针解引用、格式化字符串、整数溢出/截断等内存安全漏洞。每条内存安全结论同样必须有具体代码位置和调用链支撑。

### 认定分级体系

| 档位 | 条件 | 输出文件前缀 |
|---|---|---|
| **确认有漏洞** | 找到实际的漏洞代码路径（如路径拼接未校验 `../`），且确认无充分防护。必须有具体代码位置和调用链。 | VULN-{surface_stem}-{n}.md |
| **确认做了有效防护** | 有风险操作但经代码分析确认有有效防护（如拼接路径后做规范化+校验 `../`、参数化查询而非拼接 SQL）。必须指出具体防护代码。 | DISMISSED-{surface_stem}-{n}.md |
| **本身无风险** | 经分析该条目不涉及文件 I/O、命令执行、SQL 拼接、网络外呼等高风险操作，不存在漏洞攻击面。 | CLEAN-{surface_stem}-{n}.md |
| **无法确认** | 有风险操作但因信息不足以确认是否为漏洞：看不到源码（三方库/黑盒）、只找到入口但追溯链断掉、不确定校验是否能被绕过。**不能降级为 CLEAN**。 | SUSPECTED-{surface_stem}-{n}.md |

### 举证要求

注意：参数校验可能以注解方式实现（如 `@NotNull`、`@Size`、`@Pattern`、`@Valid`），不要遗漏方法体之外的校验。

- **路径穿越**：必须找到实际拼接用户输入到文件路径的代码，且未做规范化（如 `Path.normalize()`）或 `../` 校验
- **命令注入**：必须找到用户输入拼接到命令字符串/数组的代码，且未做转义或白名单
- **SQL 注入**：必须找到用户输入拼接到 SQL 语句的代码（`${}` / `+`），而非参数化绑定（`#{}` / `?`）
- 其他类型类推

非漏洞场景参考：`references/non-vuln-scenarios.md`

### 信息不足时的处理

- 追溯链断在三方库边界 → 结合函数名和上下文判断：函数名和参数已能明确推断其行为的，即使未找到定义也按正常处理，无需过度假设存在安全风险；仅函数名或参数路径存在高危操作嫌疑时，才记为 **疑似**（标注断点位置）
- 未找到定义的函数如果功能是获取命令路径或文件根路径（如 `getScriptDir`、`getRootPath`、`getBaseDir`、`getHome` 等含 path/dir/root/home 关键词的函数），此类路径一般为后台写死、非外部可控，按无风险处理
- 存在高危操作但输入源不确定来自外部用户 → 结合变量名和上下文判断：变量名表明是后台内部配置（如 `SPRING_*`、`SERVER_*`、`APP_*` 等规范配置项），且无证据表明攻击者可控制该变量来源的，按"后台写死、无篡改场景"处理；仅输入源来自外部请求参数、用户可写存储、或存在攻击者可控的配置注入路径时，才记为 **疑似**
- 看到校验但不确定是否能被绕过 → 沿调用链追校验逻辑的实现；仍无法确定的 → **疑似**（说明不确定性原因）
- 注入类漏洞输入源仅能追溯到 **数据库**、无法判断是否来自外部用户时 → 记为 **疑似**，不得擅自定为"非问题"
- **极大概率认定**：虽有信息不全（如追溯链末端的入参名类型模糊、校验实现细节不明），但从入口和上下文已能高度确信存在漏洞的（如入口参数直接拼接到高危操作且无可见防护的实质性证据），记为 **确认** 并在事实依据中注明信息缺口。不得机械套用"信息不足→疑似"。

### 复杂攻击面的子任务派发

进入分析前，根据当前攻击面的规模和复杂度智能规划子任务（是否拆分、怎么拆、怎么合由 LLM 自主判断）。

## 输出格式

### 文件命名规则

| 前缀 | 含义 |
|------|------|
| VULN-{surface_stem}-{n}.md | 确认有漏洞 |
| DISMISSED-{surface_stem}-{n}.md | 确认有防护 |
| CLEAN-{surface_stem}-{n}.md | 本身无风险 |
| SUSPECTED-{surface_stem}-{n}.md | 无法确认 |

（`surface_stem` 为 `surface_file` 去掉扩展名的部分；`n` 从 1 开始递增）

### 输出内容模板

**VULN 模板**：

```
# {漏洞名称}

**类型**：{漏洞分类}
**位置**：{文件:行号}
**CVSS 评分**：{X.X} **严重性**：{无/低/中/高/严重}
**触发条件**：{如何触发}
**影响**：{可能造成的危害}
**Payload**：{具体攻击 payload 或 PoC}
**攻击步骤**：（复杂漏洞必填）{逐一列举攻击者需要执行的操作步骤}
**调用链**：{入口 → ... → 敏感操作}
**建议**：{修复建议}
**事实依据**：{关键代码片段 + 分析说明，必须有代码位置引用和调用链}
```

**DISMISSED 模板**：

```
# {接口名} — 已排除

**风险操作**：{文件:行号} — {具体的风险操作}
**防护代码**：{文件:行号} — {具体的防护措施}
**排除原因**：{为什么防护有效，如校验了 `../`、参数化绑定等}
```

**CLEAN 模板**：

```
# {接口名} — 无风险

**分析摘要**：{该条目不涉及高风险操作的说明}
**检视范围**：{列举已确认的关键控制点/操作类型，证明无攻击面}
```

**SUSPECTED 模板**：

```
# {接口名} — 无法确认

**风险操作**：{文件:行号} — {具体的风险操作}
**信息缺口**：{具体哪段信息缺失导致无法确认}
**尝试**：{已做的分析尝试，如追溯链断点、未找到的校验逻辑等}
```

## 原则

- **严格限定范围**：只分析本任务指定的漏洞模式，禁止加载与分析任务无关的漏洞规则。**以任务为重，疑点所指即分析目标。仅分析任务定义的范围，不分析其他章节或文件中的任务。**
- **默认走 source-plan-vuln-tasks**：默认模式下，除非用户任务具体到可跳过（参考「规划与跳过」节），否则先调 source-plan-vuln-tasks 规划再分析
- **基于事实，不基于假设**：每条结论必须有代码级事实支撑
- **不得静默跳过**：所有外部可控输入必须生成独立结论文件（VULN / DISMISSED / CLEAN / SUSPECTED）
- **不替调用方决策**：不创建完成信号文件、不主动删/跳产物；幂等策略由调用方负责
- **不动源**：不修改任何源文件、配置文件、source-analyze 产物
- **按需分配**：根据用户任务和攻击面复杂度灵活处理单条或多条
- **保留产物对应关系**：输出文件名 = 输入文件名 stem + 编号后缀，仅目录从 `analyzed_surfaces/` 变成 `vuln_findings/`
- **失败显式标注**：走不通就显式标注、写进 `## 信息缺口` 或 `.vuln_agent_output/meta/error/source-analyze-vulnerability.md`
- **不动目标分析目录**：所有产物、临时文件、临时脚本**只能**写到 `.vuln_agent_output/` 下，**不得**在被分析项目源码目录里写任何文件
