# source-analyze-vulnerability

## 定位

- **用户负责**：给一个具体的暴露面产物文件（位于 `.vuln_agent_output/analyzed_surfaces/`），或描述想分析的目标（直接分析模式）
- **skill 负责**：读文件，以 `references/FALSE-rules.md` 为最高优先级（命中即跳过）；按 `references/constraints.md` 的范围约束做深入分析；按 `references/vuln_rules_index.md` 的哨兵索引**按需**加载具体规则文件；按本文「输出格式」一节的模板落盘 VULN / NOVULN / SUSPECTED 三类结论

skill 根据用户任务做漏洞分析，输入是 `analyzed_surfaces/` 下的一个或多个文件，按规模和复杂度智能分配处理，输出对应 `vuln_findings/` 下的结论文件。

## 路径约定

本 skill 涉及的所有目录**统一在 `.vuln_agent_output/` 下**，当前工作目录视为被分析项目根：

```
.vuln_agent_output/
├── analyzed_surfaces/                    ← 输入（默认模式的源文件，可含子目录）
│   └── {category}/...                   ← 如 REST/、MQ/ 等按分类分的子目录（可选）
├── vuln_findings/                        ← 输出（产物落盘位置，镜像输入子目录结构）
├── meta/
│   ├── batches/{surface_stem}/           ← 复杂暴露面子任务中间产物
│   └── error/source-analyze-vulnerability.md    ← 失败日志
└── temp/
    └── scripts/                          ← 临时脚本（用完即弃）
```

## 任务澄清

### 默认模式

用户没指定文件或指向 `analyzed_surfaces/` 下的文件：
- **默认递归处理 `analyzed_surfaces/` 下所有 `.md` 文件**，无需反问
- 同名产物默认覆盖。多文件由调用方并发派发（5 并发）
- 输入文件名带 MMDD-HHMMSS 时间戳，产物沿用输入 stem 保留时间戳；输入在子目录中的，产物保持同一子目录结构
- 用户显式给出路径/文件名 → 处理该文件，行为不变

### 直接分析模式

两种情况：无 surface 文件（用户消息里直接描述目标）或文件不在 `analyzed_surfaces/` 下（如直接给源码文件）。
直接分析模式下产物文件名由 skill 根据用户输入派生。

## 工作流程

1. **预加载**：按顺序读 `references/FALSE-rules.md`、`references/constraints.md`、`references/vuln_rules_index.md`、`references/vuln-principles.md`。未读完不允许动笔。  
   > `references/vuln_rules/*.md` 不在启动预加载范围——按需加载（仅疑点匹配索引时）。禁止一次性加载所有规则，禁止加载与当前任务无关的规则。
2. **解析任务**：从用户调用消息中提取：
   - `surface_file` / `analyzed_surface`（如已指定，业务流分析文件路径）
   - `discovered_surface`（如已指定，原始暴露面文件路径）
   - `plan_files`（如已指定，规划文件目录或列表）
   - `task_content`（本次分析任务——漏洞模式 / 规则 / 关注点 / 范围等）
3. **读原始暴露面文件**：打开 `.vuln_agent_output/discovered_surfaces/{discovered_surface}` 拿到原始入口信息（URL、参数约束、来源位置、描述、发现等）。`discovered_surface` 可含子目录路径，如 `REST/iface-REST-user-list-0608-021435.md`
4. **读业务流分析**：打开 `.vuln_agent_output/analyzed_surfaces/{analyzed_surface}` 拿到调用链、关键控制点、入参流向等
5. **读分析计划**：读取 `.vuln_agent_output/vuln_plans/{surface_stem}/` 下所有 `{优先级}-risk-{n}.md`，获取 vuln-planner 产出的优先级判定与敏感操作位置。**无 plan 文件时**：按原流程自行分析（等价于全量高优先级）
6. **识别疑点**：先根据暴露面规模和复杂度智能规划子任务（是否拆分、怎么拆、怎么合由 LLM 自主判断）。然后根据原始暴露面信息 + 业务流分析 + 分析计划识别高危操作（命令/文件/SQL/网络/反序列化）与外部可控输入的组合，每个疑点为一个独立分析单元
7. **列规则**：根据 `vuln_rules_index.md` 列出与疑点匹配的规则模式
8. **分批分析**：按优先级分批，每批不超过 3 个模式：
   - 入口 → 参数流向
   - 中间处理逻辑
   - 敏感操作槽
   - 每段分析产物写入 `.vuln_agent_output/meta/batches/{surface_stem}/`（按文件名隔离避免并行冲突）
9. **举证核查**：检查每条结论是否有代码级事实支撑
10. **Payload 迭代验证**：仅对"确认"漏洞生成 payload 并验证可行性
11. **收集所有批次结果，写入最终输出目录**

## 分析方法

### 基本规则

**核心原则：基于事实，不基于假设。** 每一项认定的背后必须有代码级的事实支撑。有校验不代表安全——必须分析校验是否存在切实可行的绕过方式。

> **分析范围说明**：当前分析仅覆盖系统的局部模块，**从数据库、消息队列等存储读取的数据**可能由其他模块或接口的外部用户写入。
>
> **URL 路径段拼接**：参数来源于 URL 路径段（path 部分，非 query/body/header）又拼接到其他 URL 的，系统框架已统一处理，不视为漏洞场景。

- **按 vuln_type 限定分析范围**：当调用方指定了 `vuln_type`（如 cmd/sql/path_traversal）时，只分析该类型的漏洞，不分析其他类型。未指定时全量分析

- **最高优先级**：`references/FALSE-rules.md` 是第一优先级。当任何描述与本文件冲突时，以 FALSE-rules.md 为准。只要命中其中的忽略场景，一律跳过，不做进一步分析。

- 每条结论必须有代码级事实支撑
- 信息不足必须用 SUSPECTED，禁止降级为 NOVULN。NOVULN 仅限本身无风险操作或已确认有效防护的条目
- **外部系统输入原则**：当代码中存在明显的网络操作（可见 IP、URL、host 等网络地址）时，从这些外部系统获取的数据视为不可信输入。仅从函数名推断不能确认为外部系统调用。
- 注入类问题 **从数据库获取的输入**，无明显证据证明安全的，按 SUSPECTED 处理
- 追踪入参到敏感操作槽的完整调用链
- **全面分析**：必须逐一检查每个请求参数的流向和每个关键控制点，不能因发现某个参数存在漏洞就停止对其他参数的分析
- **全链路数据流优先**：漏洞分析要**主要关注整个数据流和调用链**（入口 → 中间处理 → 敏感操作槽），**不能只看局部**——一处看似无害的拼接，在整条调用链的某个环节可能成为漏洞。**重点排查**参数如何在层与层之间传递、中间件/校验函数是否被绕过、敏感操作之前的最后一道防线是否真的有效。
- **注入类漏洞必出文件**：只要输入来自外部可控，不论是否有漏洞都必须生成独立的结论文件（VULN / NOVULN / SUSPECTED），禁止静默跳过
- **一接口多发现原则**：一个攻击面可对应多个独立的安全发现。不同漏洞类型、不同参数、或同参数的不同风险操作，**都必须各自生成独立的结论文件**，按 {n} 递增编号。严禁将多个不相关的发现合并到一个文件中。
- **穷举原则**：必须穷举该攻击面下所有安全风险点，不能因已发现一个漏洞就提前结束分析。每个独立发现都对应一个结论文件，不存在"找到一个就收工"的情况。
- **贴出关键代码举证**：每条结论必须贴出**关键代码片段**（`{文件路径:行号}` + 原文）作为事实依据，不得仅靠自然语言描述。**尤其当认为校验存在被绕过的问题时，必须贴出校验代码本身**（含注解式校验如 `@NotNull`、`@Size`、`@Pattern`、`@Valid`），否则不构成有效证据。
- **C/C++ 项目内存安全**：如果分析对象是 C/C++ 项目，还需关注缓冲区溢出、越界访问、释放后使用、空指针解引用、格式化字符串、整数溢出/截断等内存安全漏洞。每条内存安全结论同样必须有具体代码位置和调用链支撑。

### 举证要求

注意：参数校验可能以注解方式实现（如 `@NotNull`、`@Size`、`@Pattern`、`@Valid`），不要遗漏方法体之外的校验。

每条结论必须有调用链完整连续——从入口到敏感操作之间每个环节的代码都需引用，缺失环节要注明。

对各漏洞类型的具体要求：

- **SQL 注入**：必须找到用户输入拼接到 SQL 语句的代码（`${}` / `+`），而非参数化绑定（`#{}` / `?`）。需贴出拼接操作代码、参数化与非参数化的对比、MyBatis `$` vs `#` 的区分使用
- **命令注入**：必须找到用户输入拼接到命令字符串/数组的代码，且未做转义或白名单。需贴出传入命令拼接/构建执行的代码、Shell / No-Shell 构造的区分、`ProcessBuilder`/`Runtime.exec` 字符串拆分情况
- **路径穿越**：必须找到实际拼接用户输入到文件路径的代码，且未做规范化（如 `Path.normalize()`）或 `../` 校验。需贴出路径拼接代码、路径规范化及校验的缺失代码
- **反序列化**：必须找到反序列化操作入口（`readObject` / `ObjectInputStream` / `pickle.load` / `yaml.load` / `jsonpickle` 等），且输入源为外部可控。需贴出 `readObject` / `ObjectInputStream` 等反序列化入口、在使用的框架默认是否包含 gadget 链
- **SSRF**：必须找到 URL 拼接或外部输入传入网络请求的代码，且协议限制和地址限制缺失。需贴出网络请求（`HttpURLConnection`/`OkHttp`/`RestTemplate`/`HttpClient`/`requests`/`urllib`）中 URL 拼接的代码、协议限制缺失
- **敏感数据泄露**：必须找到敏感数据被输出到日志/响应/文件的代码位置，确认数据为密码/Token/密钥/证书私钥/手机号/邮箱等。需贴出返回或写入日志的具体数据字段、对应的代码位置
- **逻辑漏洞**：需画清业务步骤和判断条件，标注缺陷位置
- **内存安全（C/C++）**：需贴出内存操作的具体位置（调用 `memcpy`/`sprintf`/`scanf`/`sscanf`/`gets`/`read` 等的传参情况、指针运算、free/delete 后的使用等）

**校验分析**：如有校验防御需要特别分析强度——白名单 > 黑名单、禁止 `../` > 替换 `../`、正则转义 > 全拒绝；安全的校验规则应有具体代码举证。


### 信息不足时的处理

- 追溯链断在三方库边界 → 结合函数名和上下文判断：函数名和参数已能明确推断其行为的，即使未找到定义也按正常处理，无需过度假设存在安全风险；仅函数名或参数路径存在高危操作嫌疑时，才记为 **疑似**（标注断点位置）
- 未找到定义的函数如果功能是获取命令路径或文件根路径（如 `getScriptDir`、`getRootPath`、`getBaseDir`、`getHome` 等含 path/dir/root/home 关键词的函数），此类路径一般为后台写死、非外部可控，按无风险处理
- 存在高危操作但输入源不确定来自外部用户 → 结合变量名和上下文判断：变量名表明是后台内部配置（如 `SPRING_*`、`SERVER_*`、`APP_*` 等规范配置项），且无证据表明攻击者可控制该变量来源的，按"后台写死、无篡改场景"处理；仅输入源来自外部请求参数、用户可写存储、或存在攻击者可控的配置注入路径时，才记为 **疑似**
- 看到校验但不确定是否能被绕过 → 沿调用链追校验逻辑的实现；仍无法确定的 → **疑似**（说明不确定性原因）
- 注入类漏洞输入源仅能追溯到 **数据库**、无法判断是否来自外部用户时 → 记为 **疑似**，不得擅自定为"非问题"
- **极大概率认定**：虽有信息不全（如追溯链末端的入参名类型模糊、校验实现细节不明），但从入口和上下文已能高度确信存在漏洞的（如入口参数直接拼接到高危操作且无可见防护的实质性证据），记为 **确认** 并在事实依据中注明信息缺口。不得机械套用"信息不足→疑似"。

### "未找到定义但输入已被约束为安全形态"的特殊规则（信息不足时的优先级判定）

当入口参数明确存在格式约束（如 `@Pattern(regexp="[a-zA-Z0-9]+")`），且外部代码调用（追不到最终定义）虽有潜在风险但该参数已被有效约束时：
- **优先认定为无风险操作**，不用因调用链断在三方库边界而判为 SUSPECTED
- **判定前提**：约束必须足够**严格**。仅有 `@NotNull`、`@Size(max=2000)` 这种不限制字符范围的约束，不足以认定为无风险——攻击者仍可在长度范围内注入恶意字符。
- **严格约束示例**：纯字母数字正则（`[a-zA-Z0-9]+`）、数字类型（`Integer`/`int`、`long`/`Long` ）、枚举（`Enum`）类型、布尔类型（`boolean`/`Boolean`）
- **非严格约束示例**：`String` 类型 + `@NotNull`/`@Size`、仅含 `@Pattern` 但正则存在绕过的可能、通配类型（`Object`、`T`、`? extends T`）接收任意对象

## 输出格式

### 分析计划模板

读取 plan 文件后，自动填充 `{analysis_plan}`：

```
## 分析计划

{analysis_plan}
```

`{analysis_plan}` 包含：
- **跟踪路线**：完整数据流路径，写明每一步预期的函数和处理
- **关注点**：敏感操作位置（命令执行、SQL 拼接、文件 I/O 等）
- **待验证疑点**：分析中需要验证的问题列表
- **输出约定**：预计输出几个结论文件
- 若 plan 中指定了优先级，分析顺序按 high → medium → low 执行

**无 plan 文件时**：等价于该 surface 所有分析点按高优先级处理。

### 结论分级与文件命名

| 判定 | 缩写 | 适用场景 | 文件名 |
|------|------|----------|--------|
| **确认漏洞** | VULN | 有风险操作 + 无有效防护 / 可绕过。必须有完整的调用链和数据流证据 | VULN-{surface_stem}-{n}.md |
| **确认无风险** | NOVULN | ① 有风险操作但确认做了有效防护（如路径规范化校验、参数化查询），必须指出具体防护代码位置；② 本身无风险操作，该条目不涉及敏感操作 | NOVULN-{surface_stem}-{n}.md |
| **无法确认** | SUSPECTED | 有风险操作但因信息不足无法确认（三方库看不到源码、追溯链断掉、校验强度不确定）。**禁止降级为 NOVULN** | SUSPECTED-{surface_stem}-{n}.md |

（`surface_stem` 为 `surface_file` 去掉扩展名的部分；`n` 从 1 开始递增）

### 输出内容模板

**VULN 模板**：

```
# {漏洞名称}

**类型**：{漏洞分类}
**位置**：{文件:行号}
**CVSS 评分**：{X.X} **严重性**：{无/低/中/高/严重}
**触发条件**：{如何触发}
**影响**：{可能造成的危害}
**Payload**：{具体攻击 payload 或 PoC}
**攻击步骤**：（复杂漏洞必填）{逐一列举攻击者需要执行的操作步骤}
**调用链**：{入口 → ... → 敏感操作}
**建议**：{修复建议}
**事实依据**：{关键代码片段 + 分析说明，必须有代码位置引用和调用链}
```

**NOVULN 模板**：

```
# {接口名} — 无风险

**分析摘要**：{经分析该条目是否存在风险操作或已确认做了有效防护}
**检视范围**：{列举已确认的关键控制点/操作类型，证明无暴露面 / 已做防护}
**防护证据**：（有防护时必填）{有风险操作但确认做了有效防护时必填，贴出具体防护代码位置 + 代码片段}
```

**SUSPECTED 模板**：

```
# {接口名} — 无法确认

**风险操作**：{文件:行号} — {具体的风险操作}
**信息缺口**：{具体哪段信息缺失导致无法确认}
**尝试**：{已做的分析尝试，如追溯链断点、未找到的校验逻辑等}
```

## 原则

- **不动源**：不修改任何源文件、配置文件、source-analyze 产物
- **不创建完成信号文件**：幂等策略由调用方负责
- **不动目标分析目录**：所有产物、临时文件、临时脚本**只能**写到 `.vuln_agent_output/` 下，**不得**在被分析项目源码目录里写任何文件
