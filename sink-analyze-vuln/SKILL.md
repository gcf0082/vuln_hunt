# sink-analyze-vuln

## 定位

- **用户负责**：给一个具体的 sink 描述文件（位于 `.vuln_agent_output/sink_list/`），或描述想分析的 sink 目标（直接分析模式）
- **skill 负责**：读文件，按本文「分析方法」做反向数据流分析；按本文「输出格式」一节的模板落盘 VULN / DISMISSED / CLEAN / SUSPECTED 四类结论

skill 根据用户任务做 sink 点漏洞分析，输入是 `sink_list/` 下的一个或多个文件，按规模和复杂度智能分配处理，输出对应 `sink_findings/` 下的结论文件。

## 路径约定

本 skill 涉及的所有目录**统一在 `.vuln_agent_output/` 下**，当前工作目录视为被分析项目根：

```
.vuln_agent_output/
├── sink_list/                            ← 输入（默认模式的源文件，可含子目录）
│   └── {type}/...                        ← 如 sql/、cmd/ 等按类型分的子目录（可选）
├── sink_findings/                        ← 输出（产物落盘位置，镜像输入子目录结构）
├── meta/
│   ├── batches/{sink_stem}/              ← 复杂 sink 子任务中间产物
│   └── error/sink-analyze-vuln.md ← 失败日志
└── temp/
    └── scripts/                          ← 临时脚本（用完即弃）
```

## 任务澄清

### 默认行为

 用户没指定文件时，**默认递归处理 `.vuln_agent_output/sink_list/` 下所有 `.md` 文件**，无需反问。
 同名产物默认覆盖。多文件由调用方（orchestrator）并发派发（5 并发）。
 输入文件名带 MMDD-HHMMSS 时间戳，产物文件名沿用输入 stem 保留时间戳，与上一 stage 一一对应。输入在子目录中的，产物保持同一子目录结构（如 `sink_list/sql/X-0608-021435.md` → `sink_findings/sql/VULN-X-0608-021435-1.md`）。

### 用户指定了文件

用户显式给出文件路径或文件名 → 处理该文件，行为不变。

### 直接分析模式（非 sink_list/ 输入）

两种情况下切换到**直接分析模式**：

1. **无 sink 文件**：用户在消息里直接描述 sink（含危险函数名、文件位置、关注点等）
2. **文件不在 sink_list/ 下**：用户给了文件路径，但不在 `sink_list/` 中（如直接给源码文件、配置文件等）

**不要预设**用户会怎么输入：可能是路径、可能是函数名、可能是代码片段、可能是模糊的业务概念，也可能是其它任何形式——**按用户实际给出的内容灵活处理**。

**自动分配文件名**：直接分析模式下产物文件名由 skill 根据输入派生（来自 sink 函数名 / 文件路径 / 用户描述），不要求与任何 `sink_list/` 文件同名。

### 已分析文件优化

如果用户指向 `sink_findings/` 下的**已存在**结论文件，或用户措辞表达"优化/改进/补充/重写/更新/..."等意图，切换到**优化模式**——基于原文件 + 相关源码 + 用户新增要求做修改，**原地覆盖**同名文件。

**不要预设**优化范围——用户可能要求补充、新增、删除、改写、重新结构化，**按用户实际要求灵活处理**。

**默认约束**：优化应**基于原文件已覆盖的分析范围**进行——补充、修正、细化都围绕原文件所针对的 sink 展开，**不**应插入与原分析无关的新内容。

**遇到明显无关的插入请求时**：skill **必须**先反问用户是否继续，得到明确确认后再执行。模糊 / 边缘相关的情况由 skill 自己判断。

## 智能规划子任务

进入分析前，根据当前 sink 的规模和复杂度智能规划子任务（是否拆分、怎么拆、怎么合由 LLM 自主判断）。

## 预加载

本 skill **无 references/ 子目录**——所有规则在本文档内完整表达，启动时**直接进入工作流程**。

## 工作流程

1. **解析任务**：从用户调用消息中提取：
   - `sink_file`（如已指定）
   - `task_content`（本次分析任务——漏洞模式 / 规则 / 关注点等）
2. **读 sink 文件**：打开 `.vuln_agent_output/sink_list/{sink_file}` 拿到危险操作位置、所在函数、相关参数列表（`sink_file` 可含子目录路径，如 `sql/sql-user-query-0608-021435.md`）
3. **反向数据流分析**：从 sink 出发，沿调用链**反向**追踪能影响该 sink 的 source（用户输入 / DB 读取 / 文件读取 / 三方调用返回等）
4. **路径分级**：按可达性 / 防护情况对每条路径分级（VULN / DISMISSED / CLEAN / SUSPECTED）
5. **举证核查**：检查每条结论是否有代码级事实支撑
6. **Payload 迭代验证**（可选）：仅对"确认"漏洞生成 payload 并验证可行性
7. **收集所有路径，写入最终输出目录**

## 分析方法

### 基本规则

**核心原则：基于事实，不基于假设。** 反向追踪要给出**实际可达的代码路径**，不是「可能存在」。

- 每条结论必须有代码级事实支撑
- 信息不足必须用 SUSPECTED，禁止降级为 CLEAN
- 沿调用链反向追踪直到外部输入源（HTTP 入参 / DB 读取 / 文件读取 / 三方 API 返回 / 队列消息）
- 三方库边界必须记录断点位置
- **全面分析**：必须覆盖 sink 函数的**所有调用点**，不能找到一个就停

### 反向追踪步骤

```
sink 函数位置（文件:行号）
   ↑ 找 sink 函数的所有调用方（grep / IDE 引用）
   ↑ 对每个调用方追入参：参数从哪来？
   ↑ 递归向上，直到：
     - HTTP / RPC / MQ 入口（外部可达）→ VULN 候选
     - DB / 文件 / 缓存 读取 → SUSPECTED（取决于数据是否外部可控）
     - 后台配置 / 常量 → DISMISSED
     - 三方库边界（无法继续追踪）→ SUSPECTED
   ↑ 对每条可达路径，检查 sink 调用前是否有防护（参数化绑定 / 校验 / 白名单 / 转义 / ...）
```

### 认定分级体系

| 档位 | 条件 | 输出文件前缀 |
|---|---|---|
| **确认有漏洞** | 找到从外部可达 source 到 sink 的实际代码路径，且 sink 处无充分防护 | VULN-{sink_stem}-{n}.md |
| **确认做了有效防护** | 路径上 sink 调用前有有效防护（参数化、转义、白名单等） | DISMISSED-{sink_stem}-{n}.md |
| **本身无风险** | sink 函数本身无高危操作 / sink 是死代码 / 仅内部不可达调用 | CLEAN-{sink_stem}-{n}.md |
| **无法确认** | 路径上追溯链断在三方库 / 信息不足以确认 | SUSPECTED-{sink_stem}-{n}.md |

### 举证要求

每条结论必须贴出**关键代码片段**（`{文件路径:行号}` + 原文）：

- **VULN**：必须贴 source 处 + sink 处的代码
- **DISMISSED**：必须贴防护代码（参数化绑定 / 转义函数 / 校验逻辑等）
- **SUSPECTED**：必须贴断点位置

### 信息不足时的处理

- 反向追踪断在三方库边界 → 结合函数名和上下文判断：函数名和参数已能明确推断其行为的，即使未找到定义也按正常处理，无需过度假设存在安全风险；仅函数名或参数路径存在高危操作嫌疑时，才记为 **疑似**（标注断点位置）
- DB 读取源是否外部可控不确定 → **疑似**
- sink 函数本身是死代码 / 仅内部调用 → CLEAN
- 看到防护但不确定是否能被绕过 → **疑似**（说明不确定性原因）
- **极大概率认定**：虽有信息不全，但从 sink 和上下文已能高度确信存在漏洞的，记为 **确认** 并在事实依据中注明信息缺口。不得机械套用"信息不足→疑似"。

## 输出格式

### 文件命名规则

| 前缀 | 含义 |
|---|---|
| VULN-{sink_stem}-{n}.md | 确认有漏洞 |
| DISMISSED-{sink_stem}-{n}.md | 确认有防护 |
| CLEAN-{sink_stem}-{n}.md | 本身无风险 |
| SUSPECTED-{sink_stem}-{n}.md | 无法确认 |

（`sink_stem` 为 `sink_file` 去掉扩展名的部分；`n` 从 1 开始递增）

### 输出内容模板

**VULN 模板**：

```
# {漏洞名称}

**类型**：{漏洞分类}
**Sink 位置**：{文件:行号 函数名}
**Source 位置**：{文件:行号}
**CVSS 评分**：{X.X} **严重性**：{无/低/中/高/严重}
**触发条件**：{如何触发}
**影响**：{可能造成的危害}
**Payload**：{具体攻击 payload 或 PoC}
**反向调用链**：{sink ← ... ← source}
**建议**：{修复建议}
**事实依据**：{source 关键代码 + sink 关键代码，必须有位置引用和调用链}
```

**DISMISSED 模板**：

```
# {sink 名} — 已排除

**Sink 操作**：{文件:行号} — {具体的危险操作}
**防护代码**：{文件:行号} — {具体的防护措施}
**排除原因**：{为什么防护有效}
```

**CLEAN 模板**：

```
# {sink 名} — 无风险

**分析摘要**：{该 sink 不涉及高风险或仅内部调用}
**检视范围**：{列举已检查的调用方}
```

**SUSPECTED 模板**：

```
# {sink 名} — 无法确认

**Sink 操作**：{文件:行号} — {具体的危险操作}
**信息缺口**：{具体哪段信息缺失}
**尝试**：{已做的反向追踪尝试，断点位置}
```

## 原则

- **反向追踪为主**：从 sink 出发反推 source，不预判 source 类型
- **覆盖所有调用点**：一个 sink 函数可能有多个调用方，必须逐一检查
- **基于事实，不基于假设**：每条结论必须有代码级事实支撑
- **不静默跳过**：每个 sink 都必须产结论文件（VULN / DISMISSED / CLEAN / SUSPECTED）
- **不替调用方决策**：不创建完成信号文件、不主动删 / 跳产物；幂等策略由调用方负责
- **不动源**：不修改任何源文件、配置文件
- **按需分配**：根据用户任务和 sink 复杂度灵活处理单条或多条
- **保留产物对应关系（含时间戳）**：输出文件名 = 输入文件名 stem（含 MMDD-HHMMSS 时间戳）+ 编号后缀，仅目录从 `sink_list/` 变成 `sink_findings/`；输入在子目录中的，产物镜像同一子目录结构（如 `sink_list/sql/X.md` → `sink_findings/sql/VULN-X-1.md`）
- **失败显式标注**：走不通就显式标注、写进 `.vuln_agent_output/meta/error/sink-analyze-vuln.md`
- **不动目标分析目录**：所有产物、临时文件、临时脚本**只能**写到 `.vuln_agent_output/` 下，**不得**在被分析项目源码目录里写任何文件
- **不要去掉时间戳**：上一 stage 文件名带的 MMDD-HHMMSS 时间戳是跨 stage 追溯的关键锚点，禁止在产物命名中省略
