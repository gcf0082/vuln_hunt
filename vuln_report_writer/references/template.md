# [漏洞名称] — [受影响组件]

> 模板说明：以下为报告的标准结构，每段后附填写指南。生成报告时保留标题层级和字段标题，删除填写指南内容。

## 漏洞摘要

简要概述：漏洞类型、影响范围、风险等级、入口点。

**填写指南**：用 2-3 句话让读者快速理解问题本质。例："/api/query 接口未对用户输入的 cmd 参数做过滤，直接拼接到 shell 命令中执行，攻击者可构造恶意 Payload 实现任意命令执行。风险等级：Critical。"

## 漏洞详情

详细描述漏洞成因、触发条件及安全影响。说明为什么这是一个安全风险，而非设计行为。

**填写指南**：
- 解释问题发生的根本原因（如：未过滤输入、错误使用危险函数、缺乏权限校验等）
- 说明攻击者如何利用该问题
- 说明为什么当前实现不符合安全最佳实践

## 调用链分析

```
入口: path/to/file.go:23   ← 外部输入入口，标注入参名
  ↓  函数调用/参数传递
中转: path/to/file.go:45 → path/to/check.go:67
  ↓  数据流传递
危险调用: path/to/exec.go:89 → exec(cmd)   ← 具体的 sink 点
```

**填写指南**：
- 必须完整标注从外部输入入口到危险调用点的**每一跳**
- 每跳必须包含：文件名 + 行号 + 数据传递方式（参数传递/拼接/反序列化等）
- 如果同一文件有多条路径，选择最直接的路径
- 如果调用链有分支，标注关键分支条件
- 箭头 `↓` 后标注数据在这跳的变化（如"字符串拼接"、"URL 解码"、"Base64 解码"）

## PoC / Payload

```bash
# 可执行的 PoC 命令或代码
curl -X POST 'http://target/api/query' \
  -d 'cmd=whoami;id'
```

```python
# 如需要更复杂的 PoC
import requests
r = requests.post('http://target/api/query', data={'cmd': 'cat /etc/passwd'})
print(r.text)
```

**填写指南**：
- PoC 必须**可执行**，不得使用 "..." 或 "your payload here" 等占位符
- 优先用 curl 命令，简洁直接。复杂场景可附带 Python/Go 脚本
- 如果漏洞需要多步触发，分步列出
- 注明 PoC 的预期输出，方便人工验证
- 如果 PoC 依赖特定环境，先说明前提条件（如 "需登录后获取 session"）

## 影响分析

**填写指南**：

- 是否需要串联其他漏洞才能构成实际危害

## 置信度

**High / Medium / Low**

判定依据：
- 调用链完整度：（几跳、是否直接可达）
- PoC 可复现性：（是否已验证或有明确证据）
- 是否需额外假设条件：（如依赖特定环境配置）

**填写指南**：

| 证据完整度 | 置信度 |
|---|---|
| 完整调用链 + 可执行 PoC + 无需额外假设 | High |
| 完整调用链，但 PoC 依赖特定条件/配置 | Medium |
| 调用链部分可推断但缺少关键跳跃，或无法构造可执行 PoC | Low |

置信度为 Low 时，报告中必须标注"此发现需人工进一步确认"，并说明具体缺失的证据。
