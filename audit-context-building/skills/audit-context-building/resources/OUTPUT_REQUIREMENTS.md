# 输出要求

执行超细粒度分析时，Claude 必须按照函数微观结构检查清单格式组织输出。

---

## 必需结构

对**每个**分析的函数，输出必须包含：

**1. 用途（必填）**
- 清晰说明函数在系统中的角色
- 对系统状态、安全性或经济性的影响
- 至少 2-3 句话

**2. 输入与假设（必填）**
- 所有参数（显式和隐式）
- 所有前置条件
- 所有信任假设
- 每个输入必须标明：类型、来源、信任级别
- 至少 3 个假设被记录

**3. 输出与影响（必填）**
- 返回值（若无则为"void"）
- 所有状态写入
- 所有外部交互
- 所有发出的事件
- 所有后置条件
- 至少 3 个影响被记录

**4. 逐块分析（必填）**
对每个逻辑代码块，记录：
- **做什么：** 该块的作用（1 句）
- **为什么在这里：** 为什么这样排序/放置（1 句）
- **假设：** 必须成立的条件（1+ 项）
- **依赖：** 依赖的先前状态/逻辑
- **第一性原理 / 5 Whys / 5 Hows：** 每块至少应用**一种**

最低标准：
- 必须分析：所有条件分支、所有外部调用、所有状态修改
- 复杂块（>5 行）：应用第一性原理**以及** 5 Whys 或 5 Hows
- 简单块（<5 行）：至少包含"做什么" + "为什么在这里" + 1 个假设

**5. 跨函数依赖（必填）**
- 内部调用（列出全部）
- 外部调用（列出全部并附带风险分析）
- 调用此函数的函数
- 与其他函数共享的状态
- 不变量耦合（此函数的不变量如何与其他函数交互）
- 至少 3 个依赖关系被记录

---

## 质量门槛

一个完整的微观分析**必须**识别出：
- 至少 3 个不变量（每函数）
- 至少 5 个假设（所有部分合计）
- 至少 3 个风险考量（尤其针对外部交互）
- 至少 1 次第一性原理应用
- 至少 3 次 5 Whys 或 5 Hows 应用（合计）

---

## 格式一致性

- 使用 Markdown 标题：主要部分用 `**部分名称：**`
- 列表使用项目符号（`-`）
- 代码片段使用代码块（` ```javascript `）
- 引用行号：`L45`、`第 98-102 行`
- 使用 `---` 水平分割线分隔块以提升可读性
