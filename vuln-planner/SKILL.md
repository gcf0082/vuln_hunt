# vuln-planner

## 定位

- **前序**：`source-analyze` 已经将暴露面分析完毕，产物在 `analyzed_surfaces/`
- **skill 负责**：读每个 analyzed surface 文件，参考风险模式 + 漏洞规则索引，判断该攻击面存在哪些分析点，按优先级输出分析计划
- **后续**：`source-analyze-vuln` 读取 vuln_plans 产物作为 `{analysis_plan}` 按计划执行漏洞分析
- **只做规划，不做分析**：不读源码、不追踪调用链、不验证漏洞

## 路径约定

所有目录统一下在 `.vuln_agent_output/`，当前工作目录视为被分析项目根：

```
.vuln_agent_output/
├── analyzed_surfaces/              ← 输入（source-analyze 产物，可含子目录）
├── vuln_plans/                     ← 输出（本 skill 产物，镜像输入子目录结构）
│   └── {surface_stem}/
│       ├── high-risk-{n}.md
│       ├── medium-risk-{n}.md
│       ├── low-risk-{n}.md
│       └── none-risk-{n}.md
├── meta/
│   └── error/vuln-planner.md       ← 失败日志
└── temp/
    └── scripts/                    ← 临时脚本
```

若目录不存在，先递归创建。

### 输出目录处理

**目录里已有文件时**：开始规划前**必须先询问用户**，让用户在三选一里挑一种处理方式：

1. **先清空再重新生成**：删掉 `vuln_plans/` 下所有 `*.md`（保留目录本身），本次按当前规划结果重新落盘
2. **覆盖已有同名文件**：本次生成的文件直接覆盖原文件
3. **追加（不同名才生成）**：本次只生成不冲突的新文件；同名跳过

用户**没有显式选择**时停下来等用户回复，不要默认走任何一种。

## 预加载

启动时**必须**先按顺序读取以下文件（路径相对本 skill 目录），然后才能开始任务：

1. `references/constraints.md`（项目分析约束：非分析目标、临时脚本位置）
2. `references/FALSE-rules.md`（**最高优先级**——命中忽略场景的直接跳过，不生成任何分析计划）
3. `references/risk-level-patterns.md`（风险等级判定模式）
4. `references/vuln_rules_index.md`（漏洞规则索引：哨兵模式，**仅作索引**）

`references/vuln_rules/*.md` **不在启动预加载范围**——这些是**按需加载的具体规则**：

- 启动时**禁止**一次性加载该目录下所有规则文件
- 启动时**禁止**加载与当前任务无关的规则文件
- **仅当**当前 surface 文件中的描述与索引中某模式匹配时，**才加载**对应规则文件参考
- **禁止**浏览索引寻找其他分析方向——以 surface 文件描述为准

未读完以上文件不允许动笔。

## 任务澄清

### 默认行为

用户没指定文件时，**默认递归处理 `.vuln_agent_output/analyzed_surfaces/` 下所有 `.md` 文件**，无需反问。

输入文件名带 MMDD-HHMMSS 时间戳，产物目录沿用输入 stem 保留时间戳，于 vuln_plans/ 下建立同名子目录。输入在子目录中的，产物保持同一子目录结构（如 `analyzed_surfaces/REST/X-0608-021435.md` → `vuln_plans/REST/X-0608-021435/`）。

### 用户指定了文件

用户显式给出文件路径或文件名 → 只处理该文件。

### 分析目录为空

直接退出，汇报"无 analyzed surface 产物，跳过规划阶段"。

## 工作流程

1. **预加载**：读 `references/constraints.md`、`references/FALSE-rules.md`、`references/risk-level-patterns.md`、`references/vuln_rules_index.md`
2. **扫描输入**：递归扫描 `analyzed_surfaces/` 下所有 `.md` 文件
3. **逐个处理**：对每个 surface 文件重复以下步骤
4. **汇报**：处理总数 + 各优先级分布 + 每个文件路径
5. **写入完成信号**：写入 `.vuln_agent_output/.plan_done` 空文件

## 分析方法

### 最高优先级

`references/FALSE-rules.md` 是第一优先级。当任何描述与本文件冲突时，以 FALSE-rules.md 为准。只要命中其中的忽略场景，**不生成任何分析计划**，直接跳过。

### 优先级判定

**仅根据 surface 文件中的描述信息进行判断，不要扫描代码。**

1. **高优先级** — 满足以下任一条件：
   - 文件中描述了高风险操作（命令执行、SQL 注入、SSRF、路径穿越、文件上传、反序列化等），且从给定信息看**无有效校验/过滤**，初步判断疑似存在安全漏洞
   - 文件中描述了配置类问题：不安全协议、未校验证书、弱加密算法、硬编码认证凭据等
   - 匹配 `references/vuln_rules_index.md` 中的模式
   - **C/C++ 内存安全**：如果分析对象是 C/C++ 项目，且描述了内存分配/释放操作（malloc、calloc、realloc、new、delete 等），标记为高优先级分析点

2. **中优先级** — 文件中描述了高风险操作，但从给定信息看**有明显的校验/过滤**，初步判断疑似安全，需深度数据流追踪确认校验是否可绕过

3. **低优先级** — 文件中无高风险操作，但有其他低危问题（如开放重定向、信息泄露、不安全的随机数等）

### 合并与拆分原则

- 每个文件原则上只包含一个分析点
- 但当多个分析点比较简单时（如同一参数的同类风险、同一函数的多个低危问题），可合并到一个文件中以减少任务数量，合并后文件名从 1 开始编号
- 存在 high-risk 分析点才写 `high-risk-{n}.md`，存在 medium-risk 才写 `medium-risk-{n}.md`，依此类推

### 兜底规则

仅当不存在任何分析点（高/中/低）时，生成一个 `low-risk-1.md` 进行通用未知漏洞分析，避免遗漏。若已有任何分析点，则不再添加此任务。

### 跳过规则

仅当业务代码特别简单（如仅几行代码、无实质业务逻辑）时，写入 `none-risk-1.md`。

### 外部提示词优先

如果用户明确要求处理某类漏洞（如"检查反序列化"、"分析 JWT 绕过"），即使不在 risk-level-patterns 的模式清单中，也标记为对应优先级的高危分析点。

## 输出格式

### 目录结构

```
vuln_plans/{surface_stem}/
├── high-risk-1.md
├── high-risk-2.md    （多个同优先级时递增编号）
├── medium-risk-1.md
├── low-risk-1.md
└── none-risk-1.md    （仅无需分析时存在）
```

### 文件命名规则

| 前缀 | 含义 | 触发条件 |
|------|------|---------|
| `high-risk-{n}.md` | 高优先级分析点 | 存在高优先级分析点时 |
| `medium-risk-{n}.md` | 中优先级分析点 | 存在中优先级分析点时 |
| `low-risk-{n}.md` | 低优先级分析点 | 存在低优先级分析点，或无任何分析点时（兜底） |
| `none-risk-{n}.md` | 无需分析 | 仅当业务代码特别简单时 |

### 有分析点

每个分析点独立一个文件，从 1 开始编号，按优先级命名：

**high-risk-1.md**：
```markdown
# 深度分析计划 — {surface_stem} — 命令注入

**优先级**：高
**敏感操作位置**：{文件路径:行号}
**类型**：{漏洞分类}
**调用链片段**：{入口 → ... → 敏感操作（基于 surface 文件已有描述）}
**分析深度**：深度数据流追踪
```

**medium-risk-1.md**：
```markdown
# 深度分析计划 — {surface_stem} — SSRF

**优先级**：中
**敏感操作位置**：{文件路径:行号}
**校验说明**：{从 surface 文件看到的已有校验，需要确认能否绕过}
**分析深度**：深度数据流追踪
```

**low-risk-1.md**：
```markdown
# 分析计划 — {surface_stem} — 开放重定向

**优先级**：低
**敏感操作位置**：{文件路径:行号}
**分析深度**：标准分析
```

### 无任何分析点（兜底）

仅当不存在任何分析点（高/中/低）时，写入 `low-risk-1.md`：
```markdown
# 分析计划 — {surface_stem} — 通用未知漏洞分析

**优先级**：低
**任务**：对该攻击面的所有外部输入参数，追踪其到敏感操作的完整数据流，检查是否存在安全漏洞。
```

### 无需分析

仅当业务代码特别简单（如仅几行代码、无实质业务逻辑）时，写入 `none-risk-1.md`：
```markdown
# 分析计划 — {surface_stem}

**结果**：无需分析
**说明**：业务代码特别简单，无分析价值。
```

## 完成自检

- [ ] 每个 surface 文件都已评估
- [ ] 命中 FALSE-rules 的 surface 已跳过（不生成文件）
- [ ] 高/中/低优先级分析点已正确分类
- [ ] 无分析点时已生成兜底 low-risk-1.md
- [ ] 业务代码特别简单时已生成 none-risk-1.md
- [ ] 多个简单分析点已适当合并
- [ ] 文件名格式正确（`{优先级}-risk-{n}.md`）
- [ ] 输出目录结构镜像输入子目录结构
- [ ] 不存在的分析点不编造

## 下游交接

- **产物位置**：`.vuln_agent_output/vuln_plans/` 下所有 `*.md` 文件
- **完成信号**：`.vuln_agent_output/.plan_done` 空文件
- **下游消费**：`source-analyze-vuln` 读取 `vuln_plans/{surface_stem}/` 下的 plan 文件，按计划执行分析
- **幂等性**：每次运行前询问用户选择处理策略
- **不修改源**：不修改任何源文件、配置文件、analyzed_surfaces 产物
