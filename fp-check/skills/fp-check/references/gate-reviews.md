# 门控审查和裁定

在将任何 Bug 报告为漏洞之前，所有六个门控审查必须通过。在所有阶段完成后，在门控审查任务中评估以下内容：

| 门 | 标准 | 通过 | 失败 |
|---|---|---|---|
| **1. 流程** | 所有阶段已完成并有记录证据 | 每个阶段都有证据 | 阶段缺乏具体证据 |
| **2. 可达性** | 攻击者可以在漏洞处到达并控制数据 | 清晰的攻击者控制路径证据 + PoC 确认 | 无法展示攻击者控制或可达性 |
| **3. 真实影响** | 利用导致 RCE、权限提升或信息泄露 | 直接影响且具有具体场景 | 仅是运营健壮性问题 |
| **4. PoC 验证** | PoC（伪代码、可执行文件或单元测试）展示了攻击路径 | 展示了攻击者控制、触发和影响 | PoC 未能展示攻击路径或影响 |
| **5. 数学边界** | 数学分析确认脆弱条件可能发生 | 代数证明显示条件可能 | 数学证明验证阻止了它 |
| **6. 环境** | 没有环境防护完全阻止利用 | 防护措施不能消除漏洞 | 环境防护完全阻止了它 |

## 裁定格式

- **真实漏洞**：所有门控审查通过 → `BUG #N 真实漏洞 — [简要漏洞描述]`
- **误报**：任何门控审查失败 → `BUG #N 误报 — [简要拒绝原因]`

如果任何阶段验证失败，记录失败并附上证据，并继续完成所有剩余阶段。仅在所有阶段完成后才发布误报裁定。

## 裁定示例

```
BUG #3 误报 — packet_handler.c:142 中的整数下溢
  门 5（数学边界）失败：第 98 行的验证确保 packet_size >= 16，
  使得 (packet_size - header_size) >= 8。下溢在数学上不可能。
```
