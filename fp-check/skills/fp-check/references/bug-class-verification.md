# 漏洞类别特定验证

不同的漏洞类别需要不同的验证方法。在第 0 步对漏洞进行分类后，除了通用验证阶段之外，还应应用以下类别特定要求。

## 内存破坏

缓冲区溢出、堆溢出、栈溢出、越界读/写、释放后使用、双重释放、类型混淆。

**首先检查语言安全性：** 安全 Rust、Go（不带 `unsafe.Pointer`/cgo）或托管语言（Java、C#、Python）中的内存破坏几乎总是误报——类型系统或运行时阻止了它。验证代码是否在 `unsafe` 块中（Rust）、是否使用 cgo/`unsafe.Pointer`（Go）或通过 JNI/P/Invoke 调用本地代码。如果代码完全在安全子集中，拒绝内存破坏主张，除非涉及编译器 Bug 或健全性漏洞。

**验证：**

- 具体破坏了什么？（哪个对象、字段或内存区域）
- 破坏的大小和偏移量是多少？攻击者能控制它们吗？
- 破坏是有用的利用原语（任意读/写、vtable 覆写、函数指针覆写）还是仅仅是崩溃？
- 使用哪个分配器（glibc、tcmalloc、jemalloc、Windows heap）？它是否有阻止利用的加固措施？
- 对于 UAF：追踪对象生命周期——什么释放了它，什么重用了内存，攻击者能控制替换对象吗？
- 对于类型混淆：证明类型不匹配存在，并且数据的错误解释导致有用的原语。

## 逻辑漏洞

认证绕过、访问控制错误、不正确的状态转换、混淆代理、通过 API 误用的权限提升。

**验证：**

- 对照规范、RFC 或设计文档检查——不仅仅是代码。实现是否匹配预期行为？
- 映射所有状态转换。系统能否达到开发者未预料到的状态？
- 识别代码中从未执行过的隐式假设。
- 对于认证漏洞：验证**所有**认证/授权路径，不仅仅是看起来有问题的那一个。是否有二次检查捕获了它？
- 逻辑漏洞通过所有边界检查和数学证明——不要让干净的静态分析说服你它是误报。

## 竞态条件

TOCTOU、数据竞态、信号处理竞态、并发状态修改。

**验证：**

- 实际竞态窗口是什么？是纳秒还是秒？
- 攻击者能扩大窗口吗（例如，通过慢速 NFS 挂载、大分配或 CPU 争用来延迟线程）？
- 验证线程模型：哪些线程/进程实际上能并发访问此数据？
- 检查所有使用中的同步原语——互斥锁、原子操作、RCU、无锁结构。
- 对于文件系统上的 TOCTOU：攻击者能在检查和使用之间控制路径吗（符号链接竞态）？

## 整数问题

溢出、下溢、截断、符号性错误、环绕。

**验证：**

- 计算中每个点的确切整数类型和范围是什么？
- 溢出是有符号的（C/C++中的未定义行为——编译器可能利用此点）还是无符号的（定义明确的环绕）？
- 追踪整数经过所有类型转换、转换和提升。截断或符号扩展发生在哪里？
- 整数问题发生后，结果值是否实际以危险的方式使用（分配大小、数组索引、循环边界）？
- 检查编译器警告（`-Wconversion`、`-Wsign-compare`）是否标记了此问题。

## 加密弱点

弱算法、不良参数、nonce 重用、填充预言机、随机性不足、时序侧信道。

**验证：**

- 对照当前标准（NIST、IETF）和已知攻击检查参数选择。"AES-128"没问题；"DES"不行。
- 验证随机性来源。PRNG 在密码学上安全吗？它是否正确播种？
- 对于 nonce 重用：证明相同的 nonce 实际上可以在实践中使用两次，不仅仅是理论上。
- 对于时序侧信道：代码是否实际可以被能够测量时序的攻击者访问？网络抖动可能使远程时序攻击不切实际。
- 对照规范的参考实现或测试向量比较实现。

## 注入

SQL 注入、XSS、命令注入、服务端模板注入、路径遍历、LDAP 注入。

**验证：**

- 追踪攻击者输入从入口点到 sink（查询、命令、模板、文件系统路径）。沿途是否有任何净化或转义？
- 检查框架是否提供自动转义（例如参数化查询、模板自动转义）。如果是，它是否实际启用且未被绕过？
- 对于 XSS：输入落在什么上下文中（HTML 主体、属性、JavaScript、URL）？每个需要不同的转义。
- 对于路径遍历：路径在访问检查之前是否规范化？`../` 或空字节能否绕过验证？
- 测试通过所有中间处理的实际载荷传递——编码、解码和转换步骤可能中和或使载荷可用。

## 信息泄露

未初始化内存读取、错误消息泄露、时序侧信道、填充预言机。

**验证：**

- 具体泄露了什么数据？并非所有泄露都相同——泄露 ASLR 基址或金丝雀的栈泄露是关键的；泄露静态字符串毫无价值。
- 泄露的数据是否实际上对攻击者进一步利用有用（ASLR 绕过、会话令牌、加密密钥）？
- 对于未初始化内存：证明内存实际上在读取点未初始化，而不仅仅是在某些代码路径上可能未初始化。
- 对于时序侧信道：攻击者能否进行足够数量且足够精确的测量？噪声水平如何？
- 对于错误消息：错误路径是否实际到达攻击者，还是仅在服务端记录？

## 拒绝服务

算法复杂度、资源耗尽、崩溃 Bug、无限循环、内存炸弹。

**验证：**

- 资源消耗比是多少？攻击者发送 X 字节，服务器消耗 Y 资源。放大是否显著？
- 资源能否被回收（连接关闭、内存释放）还是永久耗尽？
- 对于算法复杂度：实际的最坏情况输入是什么？证明它触发了最坏情况行为，不要仅仅声称 O(n²)。
- 对于崩溃 Bug：崩溃是否可靠地触发，还是取决于特定的堆/栈布局？
- 服务是否自动重启？导致 100ms 重启的崩溃与需要手动干预的崩溃不同。

## 反序列化

不安全的反序列化、对象注入、gadget 链利用。

**验证：**

- 攻击者是否实际控制到达反序列化调用的序列化数据？
- classpath/导入图中是否存在可用的 gadget 链？没有 gadget 链，不安全的反序列化是设计味道，而非可利用的 Bug。
- 使用的是哪个反序列化库和版本？是否存在已知的 gadget 链？
- 是否存在类型限制、白名单或前瞻反序列化过滤器来阻止危险类？
- 针对特定语言：Java `ObjectInputStream`、Python `pickle`、PHP `unserialize`、.NET `BinaryFormatter` 各有不同的利用特征。
