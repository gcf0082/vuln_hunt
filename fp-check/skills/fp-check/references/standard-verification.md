# 标准验证

针对简单 Bug 的线性单遍检查表。无需创建任务——按每个步骤顺序工作，内联记录发现。

## 升级检查点

此检查表中有两个点可能触发升级到[深度验证]({baseDir}/references/deep-verification.md)：

1. **第 1 步（数据流）之后**：如果发现 3 个以上信任边界、回调/异步控制流或歧义的验证链时升级
2. **第 5 步（魔鬼代言人）之后**：如果任何问题产生你无法解决的真正不确定性时升级

升级时，移交到目前为止收集的所有证据——深度验证将从你停止的地方继续。

## 检查表

### 第 1 步：数据流

追踪数据从 source 到声称的漏洞 sink。

- 映射经过的信任边界（内部/可信 vs 外部/不可信）
- 识别 source 和 sink 之间的所有验证和净化逻辑
- 检查 API 契约——许多 API 具有内置的边界保护，可以阻止声称的问题
- 检查环境防护（编译器、运行时、操作系统、框架）是否完全阻止利用（不仅仅是提高门槛）
- 应用来自 [bug-class-verification.md]({baseDir}/references/bug-class-verification.md) 的类特定检查

**关键陷阱**：孤立分析脆弱代码。上游的条件逻辑可能使漏洞在数学上不可达。追踪完整的验证链。

**升级检查**：如果你发现路径中有 3 个以上信任边界、回调或异步控制流，或歧义的验证链——升级到深度验证。

### 第 2 步：可利用性

证明攻击者能触发该漏洞。

- **攻击者控制**：证明攻击者控制到达脆弱操作的数据。由可信组件设置的内部存储不是攻击者控制的。
- **边界证明**：对于整数/边界问题，使用 [evidence-templates.md]({baseDir}/references/evidence-templates.md) 中的模板创建显式的代数证明。验证：如果验证检查通过，则边界保证成立。
- **竞态可行性**：对于竞态条件，证明并发访问实际上是可能的。单线程初始化和同步上下文中不可能存在竞态条件。

### 第 3 步：影响

确定利用是否有真正的安全后果。

- 区分真正的安全影响（RCE、权限提升、信息泄露）和运营健壮性问题（崩溃恢复、清理失败）
- 区分主要安全控制和纵深防御。如果主要保护措施仍然完好，纵深防御措施的失效不是漏洞。

### 第 4 步：PoC 草稿

创建一个伪代码 PoC 展示攻击路径。可执行和单元测试 PoC 对于标准验证是可选的。

```
数据流：[Source] → [验证?] → [转换?] → [脆弱操作] → [影响]
攻击者控制：[什么输入，如何控制]
触发：[伪代码展示利用路径]
```

参见 [evidence-templates.md]({baseDir}/references/evidence-templates.md) 获取完整的 PoC 模板。

### 第 5 步：魔鬼代言人抽查

回答这 7 个问题。如果任何一个产生真正的确定性，升级到深度验证。

**反对漏洞：**

1. 我是因为模式"看起来很危险"而看到漏洞，还是因为它真的是漏洞？（模式匹配偏差）
2. 我是否错误地假设攻击者控制着可信数据？（信任边界混淆）
3. 我是否严格证明了漏洞的数学条件可以发生？（证明严谨性）
4. 我是否将纵深防御失效与主要安全漏洞混淆了？（纵深防御混淆）
5. 我是否在幻觉这个漏洞？LLM 倾向于到处看到 Bug——这实际上是真实的，还是我在对看起来吓人的代码做模式匹配？（LLM 自我检查）

**支持漏洞（始终问——假阴性防护）：**

6. 我是否因为利用看起来复杂或不太可能而忽略了真正的漏洞？
7. 我是否在编造我尚未在实际源代码中验证过的缓解措施或验证逻辑？在做结论后重新阅读代码。

**升级检查**：如果以上任何问题产生了你无法用现有证据解决的真正不确定性——升级到深度验证。

### 第 6 步：门控审查

应用 [gate-reviews.md]({baseDir}/references/gate-reviews.md) 中的所有六个门控和 [false-positive-patterns.md]({baseDir}/references/false-positive-patterns.md) 中的所有 13 项，以得出裁定。

### 第 7 步：写入输出文件

按 SKILL.md「输出要求」节将验证结果写入文件。
