# 深度验证

针对复杂 Bug 的完整基于任务验证。当从 SKILL.md 选择了深度验证路径，或标准验证升级时使用。

## 如果从标准验证升级

当 Bug 从标准验证升级时：

1. 审查标准验证过程中收集的所有证据——不要重复已完成的工作
2. 确定以下哪些阶段已被现有证据满足
3. 仅为剩余阶段创建任务，从标准验证停止的地方开始
4. 在新的任务描述中保留并引用所有先前的发现

## 验证任务列表

对于每个 Bug（Bug #N），按以下依赖结构创建任务。创建所有任务后，使用 TaskCreate 返回的任务 ID 在 TaskUpdate 中使用 `addBlockedBy` 来连接依赖。

```
── 阶段 1：数据流分析 ──────────────────────────────────
"BUG #N - 阶段 1.1：映射信任边界并追踪数据流"
  然后并行（每个被 1.1 阻塞）：
  "BUG #N - 阶段 1.2：研究 API 契约和安全保证"
  "BUG #N - 阶段 1.3：环境防护分析"
  "BUG #N - 阶段 1.4：交叉引用分析"

── 阶段 2：可利用性验证（被阶段 1 阻塞）───────────────
  并行：
  "BUG #N - 阶段 2.1：确认攻击者控制输入数据"
  "BUG #N - 阶段 2.2：数学边界验证"
  "BUG #N - 阶段 2.3：竞态条件可行性证明"
  然后（被 2.1、2.2、2.3 阻塞）：
  "BUG #N - 阶段 2.4：对抗性分析"

── 阶段 3：影响评估（被阶段 2 阻塞）─────────────────────
  并行：
  "BUG #N - 阶段 3.1：展示真正的安全影响"
  "BUG #N - 阶段 3.2：主要控制 vs 纵深防御"

── 阶段 4：PoC 创建（被阶段 3 阻塞）────────────────────
  "BUG #N - 阶段 4.1：创建带数据流图的伪代码 PoC"
  然后并行（每个被 4.1 阻塞）：
  "BUG #N - 阶段 4.2：如可行，创建可执行 PoC"
  "BUG #N - 阶段 4.3：如可行，创建单元测试 PoC"
  "BUG #N - 阶段 4.4：反面 PoC——展示利用前提条件"
  然后（被 4.2、4.3、4.4 阻塞）：
  "BUG #N - 阶段 4.5：验证 PoC 展示了该漏洞"

── 阶段 5：魔鬼代言人（被阶段 4 阻塞）───────────────────
  "BUG #N - 阶段 5.1：魔鬼代言人审查"

── 门控审查（被阶段 5 阻塞）──────────────────────────────
  "BUG #N - 门控审查：在裁定前评估所有六个门"
```

## 执行规则

- 每个任务开始时标记为进行中，只有在有具体证据时才标记为完成
- **并行子阶段**：使用插件的代理并发启动独立的子阶段。在进入下一个依赖门之前收集所有结果。
- **依赖门控**：在所有依赖任务完成之前，不要开始一个阶段。
- 对每个 Bug 应用 [false-positive-patterns.md]({baseDir}/references/false-positive-patterns.md) 中的所有 13 项检查表

## 代理

通过 `Task` 为各自阶段生成这些代理。传递 Bug 描述和任何先前阶段的结果作为上下文。

| 代理 | 阶段 | 目的 |
|---|---|---|
| `data-flow-analyzer` | 1.1–1.4 | 追踪数据流，映射信任边界，检查 API 契约和环境防护 |
| `exploitability-verifier` | 2.1–2.4 | 证明攻击者控制、数学边界、竞态条件可行性 |
| `poc-builder` | 4.1–4.5 | 创建伪代码、可执行、单元测试和反面 PoC |

阶段 3（影响评估）、阶段 5（魔鬼代言人）和门控审查直接处理——它们需要跨阶段综合结果，不应委托。

## 阶段要求

上面的任务列表命名了每个阶段。以下是每个阶段的关键陷阱和决策标准——重点关注你可能出错的地方。

### 阶段 1：数据流分析

**1.1**：映射信任边界（内部/可信 vs 外部/不可信）并追踪数据从 source 到声称的漏洞。应用来自 [bug-class-verification.md]({baseDir}/references/bug-class-verification.md) 的类特定验证。**关键陷阱**：不追踪完整验证链就孤立分析代码。上游的条件逻辑可能使脆弱代码在数学上不可达（参见 [false-positive-patterns.md]({baseDir}/references/false-positive-patterns.md) 第 1 项和第 1a 项）。

**1.2**：在声称溢出之前检查 API 契约——许多 API 具有内置的边界保护，无论输入如何都能阻止声称的问题。

**1.3**：在得出漏洞结论之前，验证没有编译器、运行时、操作系统或框架保护措施阻止利用。注意：像 ASLR 和栈金丝雀这样的缓解措施提高了利用门槛，但并未消除漏洞本身。区分"完全阻止利用"（例如 Rust 的安全类型系统）和"使利用更难"（例如 ASLR）。

**1.4**：检查是否存在类似的代码模式并在其他地方安全地处理。审查此区域的测试覆盖、代码审查历史以及设计文档。

### 阶段 2：可利用性验证

**2.1**：证明攻击者控制到达漏洞的数据。**关键陷阱**：假设网络/外部数据到达该操作而不追踪实际路径——由可信组件设置的内部存储不是攻击者控制的。

**2.2**：为边界相关问题创建显式的代数证明。使用 [evidence-templates.md]({baseDir}/references/evidence-templates.md) 中的模板。验证：如果验证检查通过，则边界保证成立。

**2.3**：对于竞态条件，证明并发访问实际上是可能的。**关键陷阱**：假设单线程初始化或同步上下文中存在竞态条件。

**2.4**：评估完整攻击面：输入控制、验证绕过路径、时序依赖和状态操纵。

### 阶段 3：影响评估

**3.1**：区分真正的安全影响（RCE、权限提升、信息泄露）和运营健壮性问题。

**3.2**：区分主要安全控制和纵深防御。如果主要保护措施仍然完好，纵深防御措施的失效不是漏洞。

### 阶段 4：PoC 创建

**始终创建伪代码 PoC。** 此外，在可行时创建可执行和/或单元测试 PoC：

1. **带数据流图的伪代码**展示攻击路径（始终）
2. **可执行 PoC** 用目标语言展示漏洞（如可行）
3. **单元测试 PoC** 用精心构造的输入锻炼脆弱代码路径（如可行）

参见 [evidence-templates.md]({baseDir}/references/evidence-templates.md) 获取 PoC 模板。

**反面 PoC（阶段 4.4）**：展示正常操作和利用路径之间的差距——触发漏洞必须满足哪些前提条件，以及为什么这些条件在正常情况下不成立。

### 阶段 5：魔鬼代言人审查

在最终裁定之前，系统性地质疑漏洞主张。假设你偏向于发现 Bug 并将其评为严重——主动对抗这种偏见。

**反对漏洞的质疑：**

1. 这个代码模式有哪些非漏洞的解释？
2. 原始开发者会如何辩护这个实现？
3. 可能缺少哪些关键的系统架构上下文？
4. 我是因为模式"看起来很危险"而看到漏洞，还是因为它真的是漏洞？
5. 即使验证看起来不充分，它是否实际上阻止了声称的条件？
6. 我是否错误地假设攻击者控制着可信数据？
7. 我是否严格证明了漏洞的数学条件可以发生？
8. 除了理论上的可能性，这在实践中是否可利用？
9. 我是否将纵深防御失效与主要安全漏洞混淆了？
10. 哪些编译器/运行时/操作系统保护可能阻止利用？
11. 我是否在幻觉这个漏洞？LLM 倾向于到处看到 Bug 并将每个发现评为严重——这实际上是一个真实的可利用问题，还是我在对看起来吓人的代码做模式匹配？

**支持漏洞的质疑（假阴性防护）：**

12. 我是否因为利用看起来复杂或不太可能而忽略了真正的漏洞？
13. 我是否在编造我尚未在实际源代码中验证过的缓解措施或验证逻辑？在做结论后重新阅读代码。

参见 [evidence-templates.md]({baseDir}/references/evidence-templates.md) 获取魔鬼代言人文档模板。
