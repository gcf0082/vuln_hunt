# 误报模式——经验教训

在验证过程中，对**每个**潜在 Bug 应用此检查表中的**所有**项。

## 检查表

### 1. 追踪完整验证链

不要分析孤立的代码片段。向后追溯，找到所有在潜在危险操作之前的验证。网络数据包大小操作可能看起来危险，但通常在该函数的前面有边界验证。

### 1a. 映射完整的条件逻辑流

看起来脆弱的代码可能由于创建数学保证的条件逻辑而不可达。示例：数组访问 `buffer[length-4]` 在 `length < 4` 时看起来不安全，但如果由于早期验证，代码仅在 `length > 12` 时才可达，则该漏洞是不可能的。

**验证：**

- 执行必须满足哪些条件才能到达声称的漏洞？
- 这些条件是否在数学上阻止了漏洞场景？
- 是否有保证安全访问的最小大小/长度要求？
- 条件流是否创建了不可违反的边界？

### 2. 识别防御性编程模式

区分实际漏洞和防御性断言/验证。`ASSERT(size == expected_size)` 后跟大小控制的操作是防御性的，不是脆弱的。验证检查是否实际阻止了声称的漏洞。

### 3. 确认可利用的数据路径

仅报告具有**已确认**可利用数据流路径的漏洞。不要假设网络控制的数据能到达危险函数而不逐步追踪实际路径。

### 4. 理解数据源上下文

区分数据源及其信任级别。API 返回值、编译时常量和网络数据具有不同的风险特征。确定实际来源以及是否由攻击者控制。

### 5. 分析边界验证逻辑

寻找验证检查和后续操作之间的数学关系。如果检查了 `packet_size >= MIN_SIZE` 且 `MIN_SIZE >= sizeof(header)`，则 `packet_size - sizeof(header)` 不可能下溢。

### 6. 验证 TOCTOU 主张

检查时间-检查-使用问题需要证明被检查的值可以在检查和使用的间隔期间改变。如果大小在同一函数中检查后立即使用，且没有外部修改的可能，则不存在 TOCTOU。

### 7. 理解 API 契约和信任边界

在声称缓冲区溢出之前始终理解 API 契约。一些 API 具有内置的边界保护，无论输入参数如何都不能写入缓冲区之外。

### 8. 区分内部存储和外部输入

内部存储系统（配置存储、注册表）由可信组件控制，而非攻击者。安装期间由可信组件设置的值不是攻击者控制的。

### 9. 不要混淆模式识别与漏洞分析

"看起来脆弱"的代码模式可能由于上下文和 API 契约而安全实现。修改大小参数不意味着缓冲区溢出，如果 API 阻止写入超出边界。

### 10. 验证并发访问实际上是可能的

不要在没有证明并发访问模式的情况下假设竞态条件存在。单线程初始化上下文中不可能存在竞态条件。验证线程模型和同步机制。

### 11. 评估真实 vs 理论安全影响

关注具有实际安全影响的漏洞。非关键数据的存储失败是运维问题，而不是安全漏洞。问：这会导致代码执行、权限提升或信息泄露吗？

### 12. 理解纵深防御 vs 主要控制

如果存在主要保护措施，纵深防御机制的失效不总是漏洞。如果令牌由设计在服务端是单次使用的，令牌清理失败不是关键问题。

### 13. 严格应用检查表，而非表面地

拥有检查表并不能防止误报，除非你系统地应用它。对于**每个**潜在漏洞，在做结论之前完成**所有**检查表中的项目。

## 误报的红旗标志

### 基于模式匹配的误报

- 在验证/边界检查代码本身中报告漏洞
- 声称 TOCTOU 而不证明值可以改变
- 忽略前面的验证逻辑
- 假设网络数据能到达操作而不追踪路径
- 混淆防御性编程（断言/检查）与漏洞
- 分析看起来脆弱的模式而不追踪控制可达性的条件逻辑
- 在错误处理或清理代码中报告"漏洞"
- 在不理解数学约束的情况下标记大小计算
- 识别"危险"函数而不检查输入是否有界
- 在具有编译时边界的固定大小操作中声称缓冲区溢出
- 在单线程或同步上下文中报告竞态条件

### 缺乏上下文分析的误报

- 分析代码片段而不理解更广泛的系统设计
- 忽略架构保证（单写入者、可信输入源）
- 未注意到"脆弱"代码由于早期验证而不可达
- 混淆调试/开发代码路径与生产路径
- 报告仅在可信安装/设置期间运行的代码中的问题
- 标记由于系统架构不可能发生的理论问题
- 未注意到声称的漏洞被框架或语言保证所阻止
- 将仅在测试或调试代码路径中的问题报告为生产漏洞

### 数学/边界分析误报

- 报告整数下溢而不证明数学条件可以发生
- 声称缓冲区溢出，而边界由验证在数学上保证
- 未注意到条件逻辑创建了脆弱条件在数学上不可能发生
- 报告离一错误而不检查循环边界是否阻止该条件
- 声称内存破坏，而分配大小已被验证足够
- 报告算术溢出而不检查输入范围是否阻止该条件

### API 契约误解误报

- 声称缓冲区溢出而 API 具有内置边界检查
- 报告使用安全管理其自身内存的 API 的内存破坏
- 未注意到返回值已被 API 契约验证
- 混淆 API 参数修改与漏洞，而 API 阻止不安全修改
- 报告被 API 安全保证显式处理的问题
- 未注意到看似危险的操作由于 API 实现细节是安全的
