# 证据模板

在记录每个 Bug 的验证证据时使用以下模板。

## 数据流文档

```
Bug #N 数据流分析
Source: [确切位置] — Trust Level: [可信/不可信]
路径: Source → 验证1[文件:行号] → 转换[文件:行号] → 漏洞[文件:行号]
验证点:
  - 检查1: [条件] 于 [文件:行号] — [通过/失败/被绕过]
  - 检查2: [条件] 于 [文件:行号] — [通过/失败/被绕过]
```

## 数学边界证明

```
Bug #N 数学分析
主张: 操作 X 易受 [溢出/下溢/边界违反] 影响
给定约束: [列出所有验证条件]

代数证明:
1. [来自验证的第一个约束]
2. [常量或已知值]
3. [推导出的不等式]
...
N. 因此: [漏洞已确认/被推翻] (证毕)

结论: [漏洞在数学上可能/不可能]
```

**示例：**

```
给定: 验证确保 (input_size >= MIN_SIZE)
给定: MIN_SIZE = 16, header_size = 8
证明: (input_size - header_size) 不可能下溢

1. input_size >= MIN_SIZE             (来自验证)
2. MIN_SIZE = 16                      (常量)
3. header_size = 8                    (常量)
4. input_size >= 16                   (代入 1,2)
5. input_size - 8 >= 16 - 8          (两边减去 header_size)
6. input_size - header_size >= 8     (简化)
7. 因此: 下溢不可能                 (证毕)
```

## 攻击者控制分析

```
Bug #N 攻击者控制分析
输入向量: [攻击者如何提供输入]
控制级别: [完全/部分/无]
约束: [攻击者输入存在哪些限制]
可达性: [攻击者控制的数据能到达漏洞操作吗？]
```

## PoC——带数据流图的伪代码

```
Bug #N 的 PoC：[简要描述]

数据流图：

[外部输入] → [验证点] → [处理] → [脆弱操作]
     |            |           |            |
  攻击者      (可能被绕过)  (转换数据)   (不安全操作)
  控制            |           |            |
     |            v           v            v
  [恶意数据] → [不充分检查] → [已处理数据] → [影响]

伪代码：
function vulnerable_operation(user_data):
    validation_result = weak_validation(user_data)  // 解释为什么这失败
    processed_data = transform_data(user_data)      // 展示转换
    unsafe_operation(processed_data)               // 展示漏洞触发
```

## 魔鬼代言人审查

```
Bug #N 魔鬼代言人审查
漏洞主张：[简要描述]

针对魔鬼代言人审查中的 13 个问题中的每一个，记录你的答案：
1-11. [反对漏洞的质疑]
12-13. [支持漏洞的质疑——假阴性防护]

最终评估：[漏洞已确认/被推翻，附推理]
```
