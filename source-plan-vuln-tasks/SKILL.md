# source-plan-vuln-tasks

## 定位

- **用户负责**：给一个具体的攻击面分析文件（位于 `analyzed_surfaces/`）
- **skill 负责**：从分析文件中识别「高危操作 + 外部可控输入」组合，划分为可独立执行的漏洞分析任务，写入 `planned_vuln_tasks/`

skill 是**单条目任务规划执行器**——一次调用只规划一个攻击面，输入是 `analyzed_surfaces/` 下的单个文件，输出是对应 `planned_vuln_tasks/` 下的若干任务文件（或一个 _no_tasks 文件）。

## 路径约定

本 skill 涉及的所有目录**统一在 `.vuln_agent_output/` 下**，当前工作目录视为被分析项目根：

```
.vuln_agent_output/
├── analyzed_surfaces/                  ← 输入
├── planned_vuln_tasks/                 ← 输出
└── meta/
    └── error/source-plan-vuln-tasks.md        ← 失败日志
```

## 任务澄清

### 何时反问

用户显式调用 `source-plan-vuln-tasks` 但**未指定 `analysis_file`** 时，**必须**先反问：

> 分析 `analyzed_surfaces/` 下的所有文件，还是指定文件？

- 用户给了文件 → 直接进入工作流程
- 用户说"全部" → 依次处理每个 analysis（**本 skill 仍每次只处理一个**，多 analysis 由调用方并发派发）

## 预加载

本 skill **无 references/ 子目录**——所有规则在本文档内完整表达，启动时**直接进入工作流程**。

## 智能规划子任务

进入规划前，根据当前 analysis 的规模和复杂度智能规划子任务（是否拆分、怎么拆、怎么合由 LLM 自主判断）。

## 工作流程

1. **解析任务**：从用户调用消息中提取 `analysis_file`
2. **读分析文件**：打开 `.vuln_agent_output/analyzed_surfaces/{analysis_file}` 拿到入口、参数、关键控制点、入参流向
3. **识别疑点**：高危操作（命令 / 文件 / SQL / 网络 / 反序列化） × 外部可控输入
4. **划分任务**：按"一个疑点 → 一个任务"拆分（同参数流向不同敏感操作必须拆开）
5. **落盘产物**：
   - 有任务 → `planned_vuln_tasks/{surface_stem}-{n}.md`（n 从 1 起）
   - 无任务 → `planned_vuln_tasks/_no_tasks-{surface_stem}.md`

## 分析方法

### 基本规则

- **不追踪源码**：只基于分析文件的描述生成任务，**不读源码、不深入追踪调用链**
- **外部可控定义**：不限于 HTTP 参数等进程外直接输入；从数据库、消息队列等存储读取的数据同样可能携带恶意内容
- **URL 路径段拼接**：参数来自 URL 路径段（path 部分，非 query/body/header）又拼接到其他 URL 的，系统框架已统一处理，**不视为疑点**
- **疑点全覆盖**：分析对象仅为系统局部，分析文件中**所有**可疑点都必须生成独立任务；不得以"该接口有多个可疑参数"为由合并为一个任务概述了之
- **任务不重复**：任务之间**不重叠同一段调用链 / 同一段代码 / 同一组参数流向**
- **同一参数流向多个敏感操作必须拆开**：禁止把不同参数的不同流向混为一谈

### 任务合并规则

"合并"**仅适用于真正相关的场景**：

- 同一敏感操作、同一调用链上的相邻参数 → 可合并
- 相同或关联的参数和操作 → 可合并
- 配置类安全问题（CORS 不当 / 认证配置缺失 / 日志配置泄漏 / TLS 弱）→ **独立**任务

### C/C++ 例外

如果分析文件的来源路径含 `.c` / `.cpp` / `.cc` / `.cxx` / `.h` / `.hpp` / `.hxx` 等后缀，**无论分析文件是否提及内存操作**，始终额外规划一个**内存安全分析任务**（缓冲区溢出 / 越界访问 / 释放后使用 / 格式化字符串 / 整数溢出 / 截断），优先级**高**。

## 输出格式

### 有任务时

每个任务写入 `.vuln_agent_output/planned_vuln_tasks/{surface_stem}-{n}.md`（n 从 1 起）：

```markdown
# {任务标题}

**验证目标**：{命令注入 / 路径穿越 / SQL 注入 / ...}
**疑点位置**：{文件路径:行号}
**疑点原因**：{简要说明为什么是疑点}
**优先级**：高/中/低
```

### 无任务时

经评估无需生成任务的，写入 `.vuln_agent_output/planned_vuln_tasks/_no_tasks-{surface_stem}.md`。逐条排查每个操作点（命令执行 / 文件 I/O / 网络 / 数据库等），每个操作点以独立条目列出：

```markdown
# {surface_stem} — 无需生成漏洞分析任务

## 总览

{总结说明为什么该攻击面无需生成分析任务}

## 逐项核查

### 1. {操作类型} — {文件路径:行号}

**操作**：{具体的命令执行 / 文件 I/O / 网络 / 数据库等操作描述}
**代码片段**：
```{语言}
{代码原文，保持完整不截断}
```
**参数来源**：{参数来自何处，如后台配置 / 常量 / 其他模块输出 / 外部输入等}
**无风险原因**：{为什么该操作不存在被利用的可能}
**举证依据**：{引用分析文件中的事实，说明自己的判断依据}
```

## 原则

- **不追踪源码**：只基于分析文件的描述生成任务
- **一个疑点 → 一个任务**：合并仅限真正相关的场景
- **任务不重复**：避免重复劳动和结论冲突
- **同参数多敏感操作必须拆**：禁止把不同流向混为一谈
- **不替调用方决策**：不创建完成信号文件、不主动删 / 跳产物；幂等策略由调用方负责
- **不动源**：不修改任何源文件、配置文件、source-analyze 产物
- **失败显式标注**：走不通就显式标注，写进 `.vuln_agent_output/meta/error/source-plan-vuln-tasks.md`
