# sink-vuln-confirm：自底向上回溯式漏洞确认

## 何时使用本 skill

你手上已经有一个**具体的可疑点**（`file:line` + 哪条语句/哪个 API 让你觉得危险），需要判定：
- **漏洞**：外部输入可控且未被充分约束，可被利用。
- **非漏洞**：每条数据通路都被与 sink 语义匹配的约束阻断。
- **不能确定**：部分路径信息不足，无法给出确定性结论。

如果还没有具体可疑点，先去做 sink 收集/暴露面扫描再回来。

---

## 核心思想

为什么要自底向上？

自顶向下的思维是"从入口看全局"——对于漏洞确认来说效率很低：你扫描了所有入口、翻了几十个 handler、沿着每条调用链走到末端，才发现末端不是 sink。自底向上的思维反过来：从已知的可疑点出发，反向搜索每一条"能到达这里的数据路"，每一条路只追到它被约束阻断、或追到外部输入为止。这棵反向调用树的**规模只与通向该 sink 的路径数成正比**，远小于全量攻击面。

5 条原则：

1. **自底向上，不是自顶向下。**
   - 从 sink 出发，反向走到各参数/拼接片段的上游赋值点，逐层外扩。
   - 直到每条分支都被终止（约束/常量）或追到外部输入边界。

2. **同时审视"可控性"与"语义穿透性"。**
   - 数据被外部影响 ≠ 漏洞（可能被白名单或强校验约束了）。
   - 数据是内部常量 ≠ 一定安全（如果拼接后路径被攻击者相对路径操纵）。
   - 一个分支判为非漏洞，需要两者都被否定——要么不可控，要么虽可控但被语义匹配的约束阻断。

3. **单分支闭合即可终止该分支，不必追到 main。**
   - 某条分支在回溯过程中遇到充分约束，立刻停止该分支，记入"已闭合"结论。
   - 未闭合的分支继续往上追。
   - 注意：不因"这一步不危险"就停下——要确认"这一步的输入"的上游也不危险，直到整条链闭合或追到外部输入。

4. **每一个判定都要被代码证据支撑。**
   - "被白名单约束" → 哪一行白名单判定，原文是什么。
   - "调用方传入常量" → 调用方在哪，字面量是什么。
   - "来自外部输入" → 读取点的 file:line，原文。
   - 没有代码引用的结论无效。读者无法复核的判定等于没有判定。

5. **信息不足时，显式声明 unknown，不脑补。**
   - 遇到动态分发、反射、跨进程、库内不可见、配置依赖、第三方 API 返回值形态不确定——标 unknown。
   - 写上"为什么看不下去"（在哪一行、缺什么信息）。
   - unknown 分支进入综合判定时会影响最终结论（含 unknown → 不能确定），但不影响已闭合分支的判定。
   - 硬撑追完然后幻觉编造，比诚实的 unknown 有害得多。

## 输出纪律

本 skill 的所有输出都必须是**可复核的**。判定不是一个标签，而是一份带证据链的论证：

- 你说"被约束" → 引用哪一行校验/转义/参数化，并解释为什么它对当前 sink 语义构成约束。
- 你说"来自外部" → 引用哪一行读取（HTTP 参数/文件 IO/env/argv/etc.）。
- 你说"看不下去" → 引用最后能看清的位置，说明从这里开始什么都看不到了（为什么看不到）和为什么看不到（动态分发/反射/跨进程/库内不可见）。

每一条"理由"都要对应一个 `file:line` + 原文代码片段。没有代码引用的结论一律视为未完成。

## 第 0 步：刻画 sink 的"危险面"

不是 sink 的所有参数、所有变量都值得追。先搞清楚**哪个/哪些数据才真正决定这条语句的危险性**。

### 0.1 识别注入面

同一个 API，不同参数的危险性完全不同。举例意在让你理解"判断思路"而不是背 API 清单：

- `Runtime.exec(String)` / `os.system(cmd)` / `subprocess.Popen(cmd, shell=True)` → 整条字符串若可控，就能改变 shell 解析的命令结构。**整串是注入面**。
- `Runtime.exec(String[])` / `ProcessBuilder(cmd, args...)` / `subprocess.Popen(list, shell=False)` → argv[0]（可执行路径）和其余参数的性质不同：argv[0] 被 PATH 解析可被劫持，参数本体已被数组边界约束（但仍受 argv 语义影响如 `-exec` 注入）。**注入面主要是 argv[0] 的路径部分，以及允许命名参数解释的工具的特定参数**。
- `PreparedStatement.setString(idx, val)` → 参数已被协议层边界约束，SQL 注入角度注入面不存在（但非参数化部分如表名拼接依然危险）。**表名/列名拼接部分是注入面**。
- `child_process.exec(cmd)` → 整串走 shell，同上第一条。`execFile(file, args)` → 同上第二条。
- 字符串拼接调用到 sink 时：注入面是**被拼接进去的变量片段**，常量字符串部分不需要回溯。
- `templateEngine.render(templateName, model)` → 注入面视引擎风险模式而定：模板名是否可控（路径穿越/ SSTI 入口），model 中的值是否被模板引擎求值。
- `eval(expr)` / `new Function(src)` / `__import__` / `Pickle.loads` → 整串/整字节序列是注入面。

### 0.2 产生 V₀（待回溯目标变量集）

从 sink 行的注入面中提取所有**非字面量的变量/表达式片段**作为 V₀。常量字符串直接丢弃——它们不构成注入面。

**输出示例**：

```
Sink 行:
  result = exec("ping -c 4 " + ip + " && " + userCmd)

注入面: ip, userCmd（常量部分 "ping -c 4 " 和 " && " 不追）
V₀ = { ip, userCmd }
```

## 第 1 步：函数内数据流回溯（最小循环单元）

这是整个回溯过程的基本操作。对每个目标变量 v，执行以下动作。

### 1.1 找到 v 的所有定值点

只关心 v **在被当前行使用之前**的最后一次定值。如果函数内有多个定值路径（if-else / try-catch / 赋值后覆盖），每一条都要作为独立分支。

定值点来源类型：

| 标签 | 含义 | 示例 |
|---|---|---|
| 字面量 | 直接赋常量 | `v = "localhost"` |
| 形参 | 来自函数入参 | `void foo(String v)` |
| 字段读 | 来自成员/类变量 | `v = this.host` |
| 全局读 | 来自全局变量/静态变量 | `v = AppConfig.CMD` |
| IO/外部 | 直接读外部输入 | `v = request.getParameter("ip")` |
| 调用返回值 | 来自另一个函数返回 | `v = sanitize(input)` |
| 容器元素 | 来自数组/List/Map/对象属性 | `v = map.get(key)` |
| 解构 | 来自解构赋值 | `v = parts[0]` |
| 闭包捕获 | 来自闭包环境 | lambda 中引用的外部变量 |

### 1.2 沿途约束登记

在 v 从定值点到 sink 消费点之间，可能存在各种中间操作。只记录那些**与 sink 语义相关**的约束操作：

- **校验**：白名单检查、类型校验（如 `instanceof Integer`）、正则匹配。记录校验逻辑的具体内容。
- **转义**：对 sink 语义有作用的转义（如 shell 语境下的 `quoteShellArg`、SQL 语境下的 `escapeSql`）。注意：HTML 转义、URL 编码、Base64 等对命令执行/SQL/SSTI 通常无效——这些要如实记录但注明"对当前 sink 语义无效"。
- **参数化**：将自由字符串转为结构化的 API（如从字符串拼接改成 `PreparedStatement`、从 `exec(String)` 改 `exec(String[])`）。
- **截取/转换**：`substring`、`replace`、`trim`、`split` 等——记录具体做了什么、产生了什么效果。

### 1.3 分支扩张

如果 v 有多个定值点（例如 if-else 两条路径分别赋值），**每个定值点成为一条独立分支**。分支 ID 使用 V₀ 起点名称 + 路径序号，如：

```
ip/1: if 分支中从 request.getParameter("ip") 来
ip/2: else 分支中从默认值 "127.0.0.1" 来
```

不同分支独立闭合、独立追踪。

## 第 2 步：跨函数回溯（调用链上行）

当第 1 步回溯到以下情况时，需要跳出当前函数：

### 2.1 形参 → 调用方实参

v 是当前函数的形参 → 找出**所有调用当前函数的地方**：
- 每个调用方中对应 v 的实参表达式，作为新的目标变量。
- 多个调用方意味着分支再度扩张。
- 对每个调用方，回到第 1 步（含第 1 步的"函数内回溯"）。

### 2.2 字段/全局 → 写入点

v 来自 `this.field` / `ClassName.staticField` / 全局变量 → 找出**所有向该字段/全局写入的代码位置**（直接赋值、setter、反射写入——仅能静态看到的写入点）：
- 每个写入点的右值作为新目标变量。
- 回到第 1 步。

注意：如果一个字段/全局被多个位置写入（尤其在多线程或不同模块中），必须全部列出。如果没有找到任何写入点（如反射写入、JNI 写入），将该分支标为 `unknown(字段无可见写入)`。

### 2.3 容器元素 → 写入点

v 来自 `map.get(key)` / `list.get(idx)` / `obj.prop` → 类似 2.2 逻辑：找出所有向该容器写入对应键/索引/属性的代码位置。

### 2.4 调用返回值 → 进入被调函数

v 来自 `foo(input)` 的返回值 → 进入 `foo` 函数内部，找到它的 **return 语句**，return 表达式成为新目标变量。在 `foo` 内部做第 1 步。

注意：
- 如果 foo 的入参同样来自外部（形参），需要在 `foo` 内同时追溯 return 表达式和入参形参的关系。
- 如果 foo 太大或太复杂，可以只看它的 return 语句直接依赖了哪些变量，而不深入 foo 内部所有逻辑——只要 return 路径单一且清晰。
- 如果 foo 有多个 return 点，每个 return 点成为一条子分支。

### 2.5 分支交叉引用（防循环）

当一条分支回溯过程中遇到了**已经出现在该分支历史中**的函数/字段（循环），则：
- 检查是否构成递归依赖。如果是，停止该分支并标为 `unknown(递归不可达)`。
- 如果只是重复但非循环（相同的函数被不同调用方分别调用），仍作为独立分支继续。

## 第 3 步：分支终止判定

每条分支在上溯过程中，遇到以下任意**确定性**条件即可终止。

### 3.1 确定性终止条件清单

**(a) 字面量常量**
变量来自字面量赋值（`"localhost"`、`42`、`true`），且自该赋值后未被任何外部数据覆盖。
- 终止点代码：赋值行 `file:line`。
- 为什么足够：该分支的供给已被代码固化。

**(b) 枚举 / 严格白名单成员**
变量经过枚举值检查或白名单比对，且白名单/枚举值的所有元素本身都是安全的（不包含可被利用的值）。
- 终止点代码：白名单/枚举判定行。
- 为什么足够：变量被限定在安全值的有限集合内，且该集合本身被证明安全。
- 注意：白名单值本身的语义也需要审视。例如白名单 `{"localhost", "127.0.0.1"}` 对命令注入安全，但对 SSRF 可能也并不需要穿透。

**(c) 与 sink 语义匹配的强校验**
变量经过了正则/模式校验，且校验模式对当前 sink 的注入面构成**完全的阻断**。
- 什么叫"匹配"？对 shell 注入，`^[a-zA-Z0-9_\-\.]+$` 是完全阻断（不允许任何 shell 元字符）。对 SQL 注入，单字符白名单如 `^[a-zA-Z]+$` 完全阻断。对 SSRF，`^https://trusted\.example\.com/` 加上协议校验是强约束。
- 什么叫"不匹配"？任意长度校验、字节数校验、非空校验——它们不影响注入语义。对 shell 注入做了 `replace(";", "")` 但保留 `|` `$()` 换行符 → 不匹配。
- 终止点代码：校验行。
- 为什么足够：解释该正则/校验为什么对当前 sink 构成完整阻断。

**(d) 与 sink 语义匹配的转义/参数化**
变量经过了专门的转义函数或 API 参数化，从根本上杜绝了注入。
- 转义函数需要对 sink 有效：shell 语境下使用 `shlex.quote()` / `ShellQuote` / `ProcessBuilder` 数组形态；SQL 语境下使用 `PreparedStatement` 占位符。
- 部分转义不是全封闭：对 shell 只做了 `escapeshellarg()` 但对整体命令是字符串拼接不是数组 exec——不可终止，需要继续检查上层拼接。
- 终止点代码：转义/参数化调用处。
- 为什么足够：说明该转义如何从 API 层面阻断注入。

**(e) 路径不可达**
该分支处于死代码中（条件判定常量折叠为 false、`if (false)`、已注释掉的代码、宏开关关闭的分支）。
- 终止点代码：条件判定行。
- 为什么足够：编译器/解释器不会执行到此处。

### 3.2 "假终止"——看似安全实则无效

以下操作**不构成**分支终止条件，因为它们不能阻断当前 sink 的注入面。当你看到以下操作时，如实登记在"路径约束"中，但分支**继续追踪不停止**：

| 操作 | 为什么无效 | 哪些 sink 无效 |
|---|---|---|
| HTML 实体转义 (`htmlspecialchars`, `escapeHtml`) | 只编码 HTML 元字符 `<>&"'`，对 shell/SQL/SSTI 元字符无影响 | 命令执行、SQL 注入、SSTI、路径穿越 |
| URL 编码/解码 | 对 shell 元字符不产生影响 | 命令执行 |
| `addslashes` / `magic_quotes` | 只转义引号，不影响 `;` `|` `$()` 换行等 | 命令执行、SSTI |
| `trim` / `strip` | 仅去除首尾空白 | 全部 |
| `substr` / 截取 | 截取后仍可能保留元字符 | 全部 |
| 长度校验 (`len < 100`) | 不影响注入语义 | 全部 |
| SQL `quote()` 函数 | 仅处理 SQL 语境，对下游 shell 无效 | 命令执行 |
| JSON 序列化/反序列化 | 不改变字串语义 | 全部 |
| Base64 编码/解码 | 不阻断元字符 | 全部 |

### 3.3 分支卡片的生成

每条分支终止后，压成一张卡片：

```
## 分支: ip/1
起点: sink 第 10 行的 ip 变量
终止: 已闭合
终止点: src/handler.java:15  request.getParameter("ip") → parser.parseStrict(ip, "^[0-9.]+$")
    代码:
    ```java
    if (!ip.matches("^[0-9.]+$")) { throw new ValidationException("invalid ip"); }
    ```
    为什么足够: 正则仅允许数字和点，排除所有 shell 元字符，对命令执行 sink 构成完全阻断
关键约束:
  - parser.parseStrict 正则校验 src/handler.java:15
分支判定: 非漏洞
判定理由: 外部输入的 ip 在被使用前经过 ^[0-9.]+$ 正则严格过滤，无法穿透到 exec 参数
```

## 第 4 步：分支结论与综合判定

### 4.1 单分支结论

- **漏洞分支**：追到外部输入，且全程无与 sink 语义匹配的约束。
- **非漏洞分支**：在回溯过程中被确定性终止条件阻断（含常量/白名单/强校验/参数化/不可达）。
- **unknown 分支**：追到无法继续的位置（深度耗尽/动态分发/反射/跨进程/库内不可见/字段无可见写入/递归），且未经过足约束。

### 4.2 综合判定

| 分支情况 | 最终判定 |
|---|---|
| 存在至少 1 条漏洞分支 | 漏洞 |
| 全部分支均为非漏洞分支 | 非漏洞 |
| 无漏洞分支，但有 unknown 分支 | 不能确定 |

### 4.3 最终判定必须写"为什么"

不要只贴一个标签。综合判定时要把"为什么"说清楚：

- **漏洞**："存在 N 条漏洞分支。分支 X 从外部入口 Y 经 Z 个跳点直接到达 sink，路径上仅有的约束是 A（对当前 sink 无效），另有 B 条分支已闭合。漏洞分支是最小触发条件的基础。"
- **非漏洞**："共有 N 条分支。分支 A 因常量终止、分支 B 因白名单终止、分支 C 因正则校验终止、分支 D 路径不可达……所有分支均被确定性条件阻断。并且对可能的质疑做防御性说明（见输出模板）。"
- **不能确定**："N 条分支已闭合，M 条分支为 unknown。unknown 分支中：分支 X 因深度耗尽（追到第 8 跳仍未闭合）、分支 Y 因字段无可见写入……尽管已闭合分支均安全，但由于存在信息缺失的分支，无法做出'非漏洞'的确定结论。"

## 长链路控制（防上下文膨胀）

当回溯树很深 / 分支很多时，代码片段会挤占上下文窗口，导致模型注意力分散、早期信息被遗忘、最终判定偏向近期的分支。以下策略共同防止这个问题。

### 1. 分支卡片化

每完成一条分支的追踪，立刻将其压缩为**分支卡片**（见 3.3 节格式）。卡片固定在 12 行以内。

一旦卡片形成，该分支的逐跳过程细节**不再需要留在活跃上下文中**。综合判定时只基于卡片做决定。

### 2. 深度预算

默认深度上限 **8 跳**。当一条分支从 sink 出发连续追溯 8 次（无论在第 1 步和第 2 步之间交替），仍未闭合：
- 停止继续向上追溯。
- 将该分支标为 `unknown(深度耗尽)`，并在卡片上记录"已追到 file:line，再上 N 个调用方未审计"。
- 这不是失败，这是诚实的边界声明。

为什么是 8？实践中超过 8 跳还未闭合的分支，要么真的连通了多层抽象（此时需要人力介入甄别），要么是污点已经扩散到无法静态回溯的广度。无论哪一种，继续往上追只会增加幻觉风险。8 不是硬性上限——如果某条分支前面 7 跳全是常量/简单传递，第 8 跳刚好到关键校验点，可以酌情多追 1-2 跳。但任何超过 8 跳的扩展都必须说明理由。

### 3. 引用最小化原则

每个回溯跳点引用代码 **±2 行**（包含目标行，前一行后一行）。如果确实需要更大的上下文窗口（如跨多行的链式调用、条件语句中的多行赋值），最大不超过 **±5 行**，并在分支卡片中说明扩展理由。

为什么这么严格？
- 数据流回溯只关心"这个值从哪里来"，不关心函数中其他逻辑。
- 整段函数会引入大量与当前分支无关的代码，稀释模型对有效信号的注意力。
- 一个 30 行函数中只有 1-2 行与当前分支相关，只引用这 2 行即可。

### 4. 分支优先级排序

开始回溯 V₀ 之前，先对目标变量排优先级——**先追最可能短闭合的**：

1. 字面量赋值 / 枚举常量 → 1 跳闭合
2. 形参且函数内附近有白名单/强校验 → 2-3 跳闭合
3. 形参且函数内无明显约束 → 需要跨函数
4. 字段/全局读 → 先做 quick 扫描写入点数，如果 >5 个则延后
5. 容器元素读 → 同上

大多数分支会在浅深度（1-3 跳）闭合，少数分支需要深入。先把容易的做了，上下文不会被提前塞满。

### 5. 阶段性摘要回顾

- **每闭合 3 条分支**：简要回顾已闭合分支的 ID + 结论 + 终止类型，确保仍记得全局。
- **开始综合判定前**：先列出"所有分支 ID + 结论"的完整清单，然后再逐一展开判定依据。

这是对抗注意力漂移的显式记忆刷新动作。

### 6. 饱和早停

当上下文已经被代码片段挤满、继续读更多文件只会把前 50 条证据挤出窗口时：
- 停止尝试追踪新分支。
- 已有分支中有尚未闭合的 → 全部降为 `unknown(上下文饱和)`。
- 给出能给出的最大确定性结论（通常为"不能确定"），并注明"N 条分支因上下文饱和未能完整追踪"。

诚实的 unknown > 幻觉的非漏洞。

### 7. 子代理 fallback（可选）

如果某条分支深度预估极大（比如形参来自一个底层工具函数且调用方 > 10 个），且剩余上下文预期不够支撑完整追踪：
- 将"追这条分支并生成一张分支卡片"作为子任务。
- 允许子代理用自己的上下文窗口独立追踪该分支。
- 主代理只接收最终的卡片结果。
- 使用本策略时在结论中注明"分支 X 通过子代理追踪"。

## 外部输入边界定义

下列来源均视为"外部输入"。分支追到其中任一点，就应标记为"来自外部输入"并进入漏洞/闭合判定：

| 类别 | 具体来源 |
|---|---|
| HTTP 请求 | URL 路径、查询参数、请求体、Header、Cookie、multipart 上传文件内容、Content-Type 等 |
| 文件 | 用户上传的文件内容；应用读取的用户文件（如日志、数据文件、配置文件——如果用户可写入）；临时文件 |
| 环境变量 | `getenv`、`os.environ` |
| 命令行参数 | `argv`、`args`、命令行 flags |
| 数据库 | DB 中读取的记录（如果该 DB 内容可被其他攻击面写入）；但应用自有只读配置表不算（需要区分） |
| 消息队列 | MQ 消息体、消息属性、Header |
| RPC 入参 | gRPC/Thrift/dubbo 等服务间调用的入参 |
| 反序列化 | 从网络/文件/DB 读取的序列化数据（JSON/XML/YAML/Protobuf/Pickle/custom format） |
| 第三方 API 返回值 | 外调 HTTP API、外部认证/授权服务器响应、云服务返回值 |
| IPC | Unix socket、共享内存、管道、有权限漏洞的服务间通道 |
| 不可信配置源 | 用户可控制的配置文件、远端配置中心中用户能改的键值、启动时动态加载的插件路径 |
| 网络流 | 原始 socket 读入、UDP/TCP 载荷、DNS 响应（如果未校验源） |

注意：同一个来源在不同上下文中角色不同。例如"数据库"中存储的手动维护的自有只读配置表不算外部输入；但存储的用户留言内容则是外部输入。区分的关键在于**内容是否最终受攻击者影响**。

## 常见误判模式

### 1. "看到 escape/sanitize 就放心"
任意名称含"escape"的函数不一定安全——必须验证：
- 它 escape 的字符集是否覆盖当前 sink 的注入向量。
- 调用的位置是否覆盖了所有可能的入口（比如在拼接前 escape 了每段，但拼接后又整体用了不安全的 eval）。

不引用具体代码看它对什么字符做了什么，就不算验证过。

### 2. "参数化就一定安全"
参数化 API（`PreparedStatement`、`ProcessBuilder(String[])`、`execFile`）不自动解决所有问题：
- `PreparedStatement` 不能参数化表名/列名/DESC-ASC——这些拼接到语句中仍然危险。
- `ProcessBuilder` 数组形态下 argv[0] 的可执行路径如果可控，仍可被 PATH 劫持或传入恶意脚本。
- 参数化只处理了"参数值"，没处理"结构"（表名、命令名、模板名）。

### 3. "白名单存在 = 安全"
白名单的内容本身需要检查：
- 白名单 `{"127.0.0.1", "0"}` 里 "0" 可能被 shell 解析为 fd 0（stdin），在某些语义下有利用意义。
- 白名单中的条目如果包含通配符/路径分隔符，可能被用于绕过。
- 检查白名单的**所有元素**，确认每一个都不会对当前 sink 产生危险。

### 4. "没有 ;|&` 字符 = 不能注入"
这是典型的元字符思维定势。shell 注入的元字符远不止 `;|&`：
- **换行符** `\n`（在 shell 中作为命令分隔符）
- **`$()` 和反引号** \` \`（命令替换）
- **`\\`**（行继续）
- **`#`**（注释掉后续）
- **Glob** `* ? [ ]`（可能被用于参数扩张）
- **`< >`**（重定向，某些语境下有用）
- 非 shell 注入中：换行符在 HTTP header 注入中表示 CRLF；双引号在 SQL 中表示字符串边界。

始终检查："这个校验是否阻断了当前 sink 上下文中的所有有效注入向量？"

### 5. "结论看起来对但没有引用代码"
这是最隐蔽的伪结论。模型可能写出"该值在进入 exec 之前经过了白名单校验"而不贴校验代码。读者无法复核。强制要求每个结论引用具体 `file:line` + 原文即解决此问题。

### 6. "只看一条分支就下结论"
一个变量有多个定值点就产生多条分支。只看 if 分支不看 else 分支，看到的可能是"漏洞"也可能是"不漏"，取决于你忽略了哪条。必须列出所有定值点对应的分支。else / catch / default 分支尤其容易被忽视。

## 结论依据

不少于以下要素：

- **注入面识别理由**：为什么 sink 的这个参数/片段才是真危险面。
- **分支判定理由**：逐条解释每条分支为何闭合 / 为何漏洞 / 为何 unknown。
- **综合判定逻辑**：把各分支结论合成最终判定的逻辑推演。

---

## 若为漏洞：最小触发条件

- Source 入口: <协议/方法/路径/参数名 等可达条件>
- 输入约束: <要绕过哪些前置校验、payload 应为什么形态>
- 触达 sink 的执行前提: <鉴权状态、配置开关、并发条件>

---

## 若为非漏洞：反驳常见质疑

- "万一调用方有别的入口呢？"——列出已审计的全部调用方 file:line。
- "万一校验能被绕过呢？"——给出校验代码 + 为什么对当前 sink 语义封闭。
- "万一同名字段在别处被写入呢？"——列出已审计的全部写入点 file:line；如果没有找到写入点则如实说明（标 unknown 而不是"肯定没有"）。

---

## 若不能确定：缺失信息

- 已观察到的最后位置: <file>:<line + 原文>
- 缺失类型: <动态分发 / 反射 / 跨进程 / 库内不可见 / 配置依赖 / 三方 API 行为 / 深度耗尽 / 上下文饱和 / 子代理未完成 / …>
- 需要补充的资料: <具体清单，如：需要运行时测试确认 XXX 的返回值形态、需要审计 YYY 库的 Y 版本实现、需要确认 Z 配置开关在目标环境中的值>

---

## 何时停止使用本 skill

- **没有具体可疑点**：先去做 sink 收集 / 暴露面扫描，有了具体的 `file:line` 再回来。
- **整个可疑点完全黑盒**：比如 sink 位于没有源码的三方 jar 中，没有任何可读代码。声明 unknown，不要硬猜。
- **可疑点不是 sink 类型问题**：如果你怀疑的是业务逻辑缺陷（权限绕过、竞争条件、弱随机数）、配置问题、协议设计问题，这个 skill 不适合。它只适合"一个具体的数据写入到某个具体的危险 API 或危险表达式"这种模式。
