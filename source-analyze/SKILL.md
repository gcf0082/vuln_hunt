# source-analyze

## 定位

- **用户负责**：给一个具体的攻击面文件（位于 `discovered_surfaces/`），或描述想分析的目标（直接分析模式）
- **skill 负责**：读文件，按 `references/constraints.md` 的范围约束做深入分析，按 `references/surface-format.md` 的格式规范落盘产物

skill 是**单条目分析执行器**——一次调用只分析一个攻击面，输入是 `discovered_surfaces/` 下的单个文件，输出是对应 `analyzed_surfaces/` 下的同名文件。

## 路径约定

本 skill 涉及的所有目录**统一在 `.vuln_agent_output/` 下**，当前工作目录视为被分析项目根：

```
.vuln_agent_output/
├── discovered_surfaces/                  ← 输入（默认模式的源文件，可含子目录）
│   └── {category}/...                   ← 如 REST/、MQ/ 等按分类分的子目录（可选）
├── analyzed_surfaces/                    ← 输出（产物落盘位置，镜像输入子目录结构）
├── meta/
│   └── error/source-analyze.md          ← 失败日志
└── temp/
    └── scripts/                          ← 临时脚本（用完即弃）
```

## 任务澄清

### 默认行为

 用户没指定文件时，**默认递归处理 `.vuln_agent_output/discovered_surfaces/` 下所有 `.md` 文件**，无需反问。
 同名产物默认覆盖。多文件由调用方（orchestrator）并发派发（5 并发）。
 输入文件名带 MMDD-HHMMSS 时间戳，产物文件名沿用输入 stem 保留时间戳，与上一 stage 一一对应。输入在子目录中的，产物保持同一子目录结构（如 `discovered_surfaces/REST/iface-REST-user-list-0608-021435.md` → `analyzed_surfaces/REST/iface-REST-user-list-0608-021435.md`）。

### 用户指定了文件

用户显式给出文件路径或文件名 → 处理该文件，行为不变。

### 攻击面目录为空 / 无 surface 文件

切换到**直接分析模式**——用户在消息里直接描述分析目标，产物仍写到 `analyzed_surfaces/`。

**不要预设**用户会怎么输入：可能是路径、可能是标识符、可能是代码片段、可能是模糊的业务概念，也可能是其它任何形式——**按用户实际给出的内容灵活处理**。

直接分析模式下，**保留产物对应关系**这条原则放宽——产物文件名由 skill 根据用户输入派生，不要求与任何输入文件同名。

### 已分析文件优化

如果用户指向 `analyzed_surfaces/` 下的**已存在**文件（区别于默认模式的 `discovered_surfaces/`），或用户措辞表达"优化/改进/补充/重画/更新/..."等意图，切换到**优化模式**——基于原文件 + 相关源码 + 用户新增要求做修改，**原地覆盖**同名文件。

**不要预设**优化范围——用户可能要求补充、新增、删除、改写、重新结构化，**按用户实际要求灵活处理**。

**默认约束**：优化应**基于原文件已覆盖的业务流**进行——补充、修正、细化都围绕攻击面本身的业务流展开，**不**应插入与该业务流无关的新内容。

**遇到明显无关的插入请求时**：skill **必须**先反问用户是否继续，得到明确确认后再执行。模糊 / 边缘相关的情况由 skill 自己判断。

## 预加载

启动时**必须**先读取 `references/constraints.md`（路径相对本 skill 目录），然后才能开始任务。

`references/surface-format.md` **不在启动预加载范围**——它是**输出格式参考**，在「落盘产物」**前**读取即可。

未读完 `constraints.md` 不允许动笔。

## 智能规划子任务

进入分析前，根据当前 surface 的规模和复杂度智能规划子任务（是否拆分、怎么拆、怎么合由 LLM 自主判断）。

## 工作流程

1. **预加载**：读 `references/constraints.md`
2. **读攻击面文件**：打开 `.vuln_agent_output/discovered_surfaces/{surface_file}` 拿到入口/接口的基本信息（URL、来源、参数、分类等）
3. **执行分析**：按「分析方法」一节对该 surface 内的入口/接口做调用链追踪、关键控制点描述、入参流向追踪
4. **绘制流程图**：按 mermaid flowchart TD 画法绘制
5. **落盘产物**：先读 `references/surface-format.md`，再按其格式写入 `.vuln_agent_output/analyzed_surfaces/{surface_file}`（保留子目录结构；如 `discovered_surfaces/REST/X.md` → `analyzed_surfaces/REST/X.md`）
   - **目录不存在** → 递归创建
   - **同名文件已存在** → **覆盖**（不替用户做"跳过/追加"决策；调用方在派发前应自己处理幂等性）

## 分析方法

### 基本规则

- **严格按输入分析**：输入文件描述单个接口/入口时，只分析该接口/入口本身，**不得扩大分析范围**。输入未提及的其他接口、模块或攻击面不属于本分析范围。

- **保留攻击面基本信息**：分析产物开头必须包含：
  - **`源攻击面文件`**：默认模式指向 `.vuln_agent_output/discovered_surfaces/{surface_file}`；优化模式指向被改写的 `.vuln_agent_output/analyzed_surfaces/{surface_file}`；直接分析模式标 `直接分析` + 用户输入简要描述
  - 攻击面标识信息（URL、方法、来源源码路径、分类等），确保后续分析可直接引用

- **描述事实，不做评判。**"代码做了/没做某项检查"属于事实陈述（**应当报告**，包括"未做"）；"这是否构成风险/漏洞"等属于评判（**仍禁止**）。禁用"风险/漏洞/不安全/可能被攻击"等结论性词。

- **明显风险标记**：分析过程中发现明显安全风险时在产物末尾新增 `## 明显风险` 节简要记录，标注风险类型和发现位置，不需要做深入利用验证。此标记不改变默认原则——只记录分析过程中自然发现的明显问题，不主动搜寻。特别是业务逻辑漏洞、DoS 类漏洞这类明显可感知的无需深入即可判断的风险。

- **不得遗漏、不得编造。** 未能定位的目标在产物末尾 `## 未能追溯的引用` 节集中列出，**不得静默忽略**。

### 路径与代码引用

- **写出目标路径**：文件 I/O / 命令执行 / 网络 / SQL / 加解密 节点必须写出目标本身——文件路径 / 完整命令行 / URL / 表名等，**禁止只写"写入文件 / 执行脚本 / 调用网关"等抽象表述**。动态片段用 `{变量名}` 标注并指明来源。

- **代码引用一律原样保留**——不论长度都不省略、不截断、不用 `...` / `…`。

- **硬编码绝对路径**：不直接访问，按后缀在工区找候选：先拆后缀序列（长→短），逐个搜索取最长命中；全未命中 → 进 `## 未能追溯的引用`。

- **变量值常量化**：常量来源（`static final` / `const` / 字面量赋值 / `@Value` 默认值 / 配置文件 / 枚举）代入字面量；运行时输入保留 `{var}`。

### 关键控制点检查清单

对每个控制点，需说明业务用途和校验细节，附 `{文件路径:行号}` + 代码块，例如：

```java
// src/main/java/com/acme/UploadController.java:71
ProcessBuilder pb = new ProcessBuilder("bash", "scripts/scan.sh", filename);
```

- 文件上传：后缀/MIME 白名单？大小上限？路径规范化（防 `../`）？
- 请求参数作为循环变量：参数是否直接控制循环次数／迭代方向？有无长度／范围限制？
- 解压包：在哪做了包内文件大小限制？在哪做了路径限制（防 `../` 条目）？
- 文件读取：路径含用户输入？规范化？
- 下载操作：读文件？内存生成？
- 命令执行：参数含用户输入？命令白名单？
- HTTP 外呼：TLS 校验？主机名校验？目标 URL 含用户输入？
- 复杂逻辑（含外部请求）：如何处理？
- SQL：参数化绑定（`#{}` / `?`）vs 字符串拼接（`${}` / `+ input`）
- 临时文件：如何处理？
- 加解密 / 签名：算法？密钥来源？
- 认证/鉴权（仅当存在时）：简述做了怎样的认证鉴权？
- 电话号码、邮箱：用来做什么？
- 认证凭据：用来做什么？
- **高圈复杂度**：函数中存在大量分支条件（多层 if/else / switch / 循环嵌套等）时，说明该位置及复杂度情况
- **复杂业务逻辑**：业务逻辑非常复杂时，描述该处处理什么业务逻辑及其复杂性表现（如多状态流转、规则判断、数据聚合等），附 `{文件路径:行号}`

### 流程图画法

用 mermaid flowchart TD 绘制，节点包含具体路径/命令/配置位置。判断类用 `{...}` 菱形节点 + 边标条件分支。观测优先级标记（**不是风险结论**）：

- **🔴 高**：涉及高危操作且操作目标拼接了外部用户输入
- **🟡 低**：涉及高危操作但不含外部用户输入
- 非观测点节点一律不加 emoji

节点描述中必须给出拼接后的完整字符串，标注变量来源：

- SQL：`SELECT * FROM users WHERE id = '{body.id}'`
- 路径：`/data/uploads/{body.filename}`
- 命令：`bash scripts/scan.sh {body.filename}`

### 入参追踪

- **捕获请求维度**：path / query / 关键 header（鉴权、Content-Type）/ body 字段树。注意检查参数上的注解式校验（如 `@NotNull`、`@Size`、`@Pattern`、`@Valid`），方法体之外的校验同样需要记录。每个校验必须给出代码位置（文件:行号）和对应的源代码片段。

- **入参流向**：追每个请求参数是否流入路径 / 命令 / URL / 写文件 / SQL。**必须沿调用链逐层追踪**，不能停在当前层下结论。只追到外部 SDK / 三方库边界为止。**边界处的关键函数（无法获取源码/无法继续追踪）必须记录函数名、所在文件路径和行号，不能静默跳过。**

## 输出格式

照搬 `references/surface-format.md` 模板。关键约定（不全，详见模板本身）：

- 单接口产物文件**不**包含外层"业务流讲解 / 整体在做什么 / 子功能 N"包装——只截取 `#### {METHOD} {URL}` 块，**升级为顶层 `# {METHOD} {URL} — 一句话功能`**
- 流程图自包含：输入流向、关键控制点、硬编码标注**嵌入节点标签**
- 入口节点统一用 `START([开始])`；接口 URL / 请求维度由标题与 `**请求**` 行承载
- 文件 I/O / 命令 / 外呼 / SQL 节点**必须**写出目标本身
- 长路径 / URL / 命令写全，放不下用 `<br/>` 换行而非 `...` 截断
- 路径 / 命令中的变量值能沿数据流回溯到常量的，**必须代入字面量**；`{var}` 占位**仅**保留给运行时动态输入
- 末尾 `## 未能追溯的引用` 节按需撰写（无则**略去整节**，不写"无"）
- 写日志一律忽略（信噪比低）

## 原则

- **只描述事实**：本 skill 不做风险评级、不下漏洞结论、不做利用验证。但分析过程中自然发现的明显风险可在 `## 明显风险` 节标记，无需深入分析
- **每个 surface 一份产物**：每次调用只分析一个攻击面；多 surface 由调用方并发派发
- **保留产物对应关系（含时间戳）**：输出文件名 = 输入文件名（包含 MMDD-HHMMSS 时间戳），仅目录从 `discovered_surfaces/` 变成 `analyzed_surfaces/`；输入在子目录中的，产物镜像同一子目录结构
- **不得静默跳过**：走不通就显式标注、写进 `## 未能追溯的引用`，并按需把失败原因写到 `.vuln_agent_output/meta/error/source-analyze.md`
- **不重复 discovery**：不去 `discovered_surfaces/` 之外找新对象，不为"详细"而扩范围
- **不动源**：不修改任何源文件、配置文件、discovery 产物
- **不替调用方决策**：不创建完成信号文件、不主动删/跳产物；幂等策略由调用方负责
- **不动目标分析目录**：所有产物、临时文件、临时脚本**只能**写到 `.vuln_agent_output/` 下，**不得**在被分析项目源码目录里写任何文件
- **不要去掉时间戳**：上一 stage 文件名带的 MMDD-HHMMSS 时间戳是跨 stage 追溯的关键锚点，禁止在产物命名中省略
