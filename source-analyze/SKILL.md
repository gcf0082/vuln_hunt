# source-analyze

## 定位

- **用户负责**：给一个具体的暴露面文件（位于 `discovered_surfaces/`），或描述想分析的目标（直接分析模式）
- **skill 负责**：读文件，按 `references/constraints.md` 的范围约束做深入分析，按 `references/surface-format.md` 的格式规范落盘产物

skill 是**单条目分析执行器**——一次调用只分析一个暴露面，输入是 `discovered_surfaces/` 下的单个文件，输出是对应 `analyzed_surfaces/` 下的同名文件。

## 路径约定

本 skill 涉及的所有目录**统一在 `.vuln_agent_output/` 下**，当前工作目录视为被分析项目根：

```
.vuln_agent_output/
├── discovered_surfaces/                  ← 输入（默认模式的源文件，可含子目录）
│   └── {category}/...                   ← 如 REST/、MQ/ 等按分类分的子目录（可选）
├── analyzed_surfaces/                    ← 输出（产物落盘位置，镜像输入子目录结构）
├── meta/
│   └── error/source-analyze.md          ← 失败日志
└── temp/
    └── scripts/                          ← 临时脚本（用完即弃）
```

## 任务澄清

**不要预设**用户输入的格式——可能是路径、标识符、代码片段、模糊的业务概念，**按实际内容灵活处理**。

### 默认模式

用户没指定文件或指向 `discovered_surfaces/` 下的文件：
- **默认递归处理 `discovered_surfaces/` 下所有 `.md` 文件**，无需反问
- 同名产物默认覆盖。多文件由调用方并发派发（5 并发）
- 输入文件名带 MMDD-HHMMSS 时间戳，产物沿用输入 stem 保留时间戳；输入在子目录中的，产物保持同一子目录结构
- 用户显式给出路径/文件名 → 处理该文件，行为不变

### 直接分析模式

`discovered_surfaces/` 为空或无 surface 文件时——用户在消息里直接描述分析目标，产物仍写到 `analyzed_surfaces/`。
产物文件名由 skill 根据用户输入派生，不要求与任何文件同名。

### 优化模式

用户指向 `analyzed_surfaces/` 下的已存在文件，或措辞表达"优化/改进/补充/重画/更新..."等意图时——基于原文件 + 相关源码 + 用户要求修改，**原地覆盖**。
**默认约束**：优化应基于原文件已覆盖的业务流进行，不应插入无关内容。
**遇到明显无关的插入请求时**：skill 必须先反问用户，确认后再执行。模糊/边缘相关由 skill 判断。

## 工作流程

1. **预加载**：读 `references/constraints.md` 和 `references/flow-principles.md`。未读完不允许动笔。
2. **读暴露面文件**：打开 `.vuln_agent_output/discovered_surfaces/{surface_file}` 拿到入口/接口的基本信息（URL、来源、参数、分类等）
3. **执行分析**：先根据 surface 规模和复杂度智能规划子任务（是否拆分、怎么拆、怎么合由 LLM 自主判断）。然后按「分析方法」对该 surface 内的入口/接口做调用链追踪、关键控制点描述、入参流向追踪
4. **绘制流程图**：按 mermaid flowchart TD 画法绘制
5. **落盘产物**：先读 `references/surface-format.md`，再按其格式写入 `.vuln_agent_output/analyzed_surfaces/{surface_file}`。**单接口产物文件不包含**外层"业务流讲解 / 整体在做什么 / 子功能 N"包装——只截取 `#### {METHOD} {URL}` 块，**升级为顶层 `# {METHOD} {URL} — 一句话功能`**
   - **目录不存在** → 递归创建
   - **同名文件已存在** → **覆盖**（不替用户做"跳过/追加"决策；调用方在派发前应自己处理幂等性）

## 分析方法

### 基本规则

- **严格按输入分析**：输入文件描述单个接口/入口时，只分析该接口/入口本身，**不得扩大分析范围**。输入未提及的其他接口、模块或暴露面不属于本分析范围。

- **保留暴露面基本信息**：分析产物开头必须包含：
  - **`源暴露面文件`**：默认模式指向 `.vuln_agent_output/discovered_surfaces/{surface_file}`；优化模式指向被改写的 `.vuln_agent_output/analyzed_surfaces/{surface_file}`；直接分析模式标 `直接分析` + 用户输入简要描述
  - 暴露面标识信息（URL、方法、来源源码路径、分类等），确保后续分析可直接引用

- **描述事实，不做评判。**"代码做了/没做某项检查"属于事实陈述（**应当报告**，包括"未做"）；"这是否构成风险/漏洞"等属于评判（**仍禁止**）。禁用"风险/漏洞/不安全/可能被攻击"等结论性词。

- **明显风险标记**：分析过程中发现明显安全风险时在产物末尾新增 `## 明显风险` 节简要记录，标注风险类型和发现位置，不需要做深入利用验证。此标记不改变默认原则——只记录分析过程中自然发现的明显问题，不主动搜寻。特别是业务逻辑漏洞、DoS 类漏洞这类明显可感知的无需深入即可判断的风险。

- **不得遗漏、不得编造。** 未能定位的目标在产物末尾 `## 未能追溯的引用` 节集中列出，**不得静默忽略**。
- **写日志一律忽略**：日志写入操作信噪比低，**不**纳入流程图和疑点分析

### 路径与代码引用

- **写出目标路径**：文件 I/O / 命令执行 / 网络 / SQL / 加解密 节点必须写出目标本身——文件路径 / 完整命令行 / URL / 表名等，**禁止只写"写入文件 / 执行脚本 / 调用网关"等抽象表述**。动态片段用 `{变量名}` 标注并指明来源。

- **代码引用一律原样保留**——不论长度都不省略、不截断、不用 `...` / `…`。

- **硬编码绝对路径**：不直接访问，按后缀在工区找候选：先拆后缀序列（长→短），逐个搜索取最长命中；全未命中 → 进 `## 未能追溯的引用`。

- **变量值常量化**：常量来源（`static final` / `const` / 字面量赋值 / `@Value` 默认值 / 配置文件 / 枚举）代入字面量；运行时输入保留 `{var}`。

### 关键控制点检查清单

对每个控制点，需说明业务用途和校验细节，附 `{文件路径:行号}` + 代码块，例如：

```java
// src/main/java/com/acme/UploadController.java:71
ProcessBuilder pb = new ProcessBuilder("bash", "scripts/scan.sh", filename);
```

- 文件上传：后缀/MIME 白名单？大小上限？路径规范化（防 `../`）？
- 请求参数作为循环变量：参数是否直接控制循环次数／迭代方向？有无长度／范围限制？
- 解压包：在哪做了包内文件大小限制？在哪做了路径限制（防 `../` 条目）？
- 文件读取：路径含用户输入？规范化？
- 下载操作：读文件？内存生成？
- 命令执行：参数含用户输入？命令白名单？
- HTTP 外呼：TLS 校验？主机名校验？目标 URL 含用户输入？
- 复杂逻辑（含外部请求）：如何处理？
- SQL：参数化绑定（`#{}` / `?`）vs 字符串拼接（`${}` / `+ input`）
- 临时文件：如何处理？
- 加解密 / 签名：算法？密钥来源？
- 认证/鉴权（仅当存在时）：简述做了怎样的认证鉴权？
- 电话号码、邮箱：用来做什么？
- 认证凭据：用来做什么？
- **内存分配/释放（C/C++）**：分配方式（malloc/calloc/new）、释放位置、是否存在缓冲区溢出、释放后使用、双重释放、空指针解引用？
- **敏感信息记录日志**：检查是否将明文或加密的认证凭据类信息（密码、Token、密钥、证书私钥、Cookie等）、电话号码、邮箱地址等敏感数据写入日志文件、控制台输出或错误响应。其他信息（如用户ID、用户名、昵称、地址等）不属于敏感信息，无需关注。
- **复杂逻辑**：函数存在大量分支条件（多层 if/else / switch / 循环嵌套）或复杂业务逻辑（多状态流转、规则判断、数据聚合）时，标注 `[复杂逻辑] {文件:行号}`。此类函数易藏安全缺陷，后续漏洞分析应重点审视。
- **疑似安全漏洞 / 业务逻辑漏洞 / DOS**：分析过程中如发现疑似安全漏洞、业务逻辑漏洞或 DOS 隐患，在对应控制点旁标注 `[疑似]` 并简要说明疑点。**仅标注、不深入分析**。例如：`[疑似] host 拼接后进入 Runtime.exec，可能构成命令注入面`

### 流程图画法

用 mermaid flowchart TD 绘制，节点包含具体路径/命令/配置位置。判断类用 `{...}` 菱形节点 + 边标条件分支。观测优先级标记（**不是风险结论**）：

- **🔴 高**：涉及高危操作且操作目标拼接了外部用户输入
- **🟡 低**：涉及高危操作但不含外部用户输入
- 非观测点节点一律不加 emoji

节点描述中必须给出拼接后的完整字符串，标注变量来源。

### 入参追踪

- **捕获请求维度**：path / query / 关键 header（鉴权、Content-Type）/ body 字段树。注意检查参数上的注解式校验（如 `@NotNull`、`@Size`、`@Pattern`、`@Valid`），方法体之外的校验同样需要记录。每个校验必须给出代码位置（文件:行号）和对应的源代码片段。

- **入参流向**：追每个请求参数是否流入路径 / 命令 / URL / 写文件 / SQL。**必须沿调用链逐层追踪**，不能停在当前层下结论。只追到外部 SDK / 三方库边界为止。**边界处的关键函数（无法获取源码/无法继续追踪）必须记录函数名、所在文件路径和行号，不能静默跳过。**
