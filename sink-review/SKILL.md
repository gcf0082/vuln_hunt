# sink-review

## 定位

- **用户负责**：给一个具体的 sink-based 漏洞分析结论文件（位于 `.vuln_agent_output/sink_findings/`），或描述想审查的目标（直接审查模式）
- **skill 负责**：读文件，以**挑战者**姿态审查第一轮 sink-based 分析结论（漏洞和非漏洞结论均需审查），结合业务特有信息判断结论是否合理；按本文「输出格式」一节的模板落盘 VULN / NOVULN / SUSPECTED 三类审查结论

skill 是**单条目 sink-based 漏洞审查执行器**——一次调用只审查一个 sink_finding 文件，输出是对应 `sink_reviews/` 下的一个审查结论文件（**1 对 1 映射**）。

## 路径约定

本 skill 涉及的所有目录**统一在 `.vuln_agent_output/` 下**，当前工作目录视为被分析项目根：

```
.vuln_agent_output/
├── sink_findings/                         ← 输入（待审查的第一轮 sink-based 结论，可含子目录）
├── sink_list/                             ← 必读（回溯到原始 sink 描述，可含子目录）
├── analyzed_surfaces/                     ← 可回溯（按需，第一轮分析的完整追踪过程与证据）
├── sink_reviews/                          ← 输出（审查结论落盘位置，镜像输入子目录结构）
├── meta/
│   └── error/sink-review.md ← 失败日志
└── temp/
    └── scripts/                           ← 临时脚本（用完即弃）
```

## 任务澄清

### 默认行为

 用户没指定文件时，**默认处理 `.vuln_agent_output/sink_findings/**/*.md` 下所有文件（含子目录）**，无需反问。
 同名产物默认覆盖。多文件由调用方（orchestrator）并发派发（5 并发）。
 输入文件名带 MMDD-HHMMSS 时间戳，产物文件名沿用输入 stem 保留时间戳，与上一 stage 一一对应。输入在子目录中的，产物保持同一子目录结构（如 `sink_findings/sql/VULN-X-0608-021435-1.md` → `sink_reviews/sql/VULN-X-0608-021435.md`）。

### 用户指定了文件

用户显式给出文件路径或文件名 → 处理该文件，行为不变。

### 直接审查模式

两种情况下切换到**直接审查模式**：

1. **无 sink_finding 文件**：用户在消息里直接描述审查目标（含 sink 信息、关注点、挑战方向等）
2. **文件不在 sink_findings/ 下**：用户给了文件路径但不在 `sink_findings/` 中

**不要预设**用户会怎么输入——**按用户实际给出的内容灵活处理**。

**自动分配文件名**：直接审查模式下产物文件名由 skill 根据输入派生。

### 已审查文件优化

如果用户指向 `sink_reviews/` 下的**已存在**审查结论文件，或用户措辞表达"优化/改进/补充/重写/更新/..."等意图，切换到**优化模式**——基于原文件 + 相关源码 + 用户新增要求做修改，**原地覆盖**同名文件。

**不要预设**优化范围——用户可能要求补充、新增、删除、改写、重新结构化，**按用户实际要求灵活处理**。

**默认约束**：优化应**基于原审查文件已覆盖的范围**进行——补充、修正、细化都围绕原审查所针对的 sink 结论展开，**不**应插入与原审查无关的新内容。

**遇到明显无关的插入请求时**：skill **必须**先反问用户是否继续，得到明确确认后再执行。

## 智能规划子任务

进入审查前，根据当前 sink_finding 的规模和复杂度智能规划子任务（是否拆分、怎么拆、怎么合由 LLM 自主判断）。

## 预加载

本 skill **无 references/ 子目录**——所有规则在本文档内完整表达，启动时**直接进入工作流程**。

## 工作流程

1. **解析任务**：从用户调用消息中提取 `sink_finding_file`
2. **读 sink_finding 文件**：打开 `.vuln_agent_output/sink_findings/{sink_finding_file}` 拿到 sink 位置、反向调用链、判定结论
3. **回溯 sink_list**：找到对应的原始 sink 描述文件（按 stem 匹配），拿到 sink 元数据（类型、危险操作、建议排查方向）
4. **可选回溯 analyzed_surfaces/**：如需更完整的反向追踪证据链，**按需**读取 `.vuln_agent_output/analyzed_surfaces/{analysis_file}`
5. **快速判断**：能基于现有证据直接给出结论的，直接输出；发现证据不足或存疑时再深入追踪代码核实
6. **质疑审查**：以挑战者姿态审查（详见「审查方法」一节）
7. **校验闭环验证**：sink 上下游调用链的防护必须**完整**核实，不能只看 sink 那行代码
8. **Payload 重构**：若上一轮 payload 不可实施，尝试重新构造可用的 payload
9. **收集审查结论，写入最终输出目录**

## 审查方法

### 基本规则

以挑战者的姿态审查第一轮 sink-based 分析结论（漏洞和非漏洞结论均需审查），结合业务特有信息判断结论是否合理。无法确定的记为疑似。

> **分析范围说明**：当前分析仅覆盖系统的局部模块，**从数据库、消息队列等存储读取的数据**可能由其他模块或接口的外部用户写入。

**核心验证原则**：检验反向链每一跳是否真实可达，是否有 sink 调用前的中间校验未被发现。验证时必须检查 sink 函数所在位置 + 上下游调用链 + 中间件校验 + 注解式校验 + 辅助方法。

- **文件输入**：并非所有输入来自文件都要假设文件被篡改。绝大多数文件是后台安装时就固定的，一般情况下无法篡改。除非找到充足的证据证明文件可被篡改，否则不认为有漏洞。**此原则不适用于从数据库获取的数据。**
- **环境变量**：绝大多数情况下环境变量是后台写死的。除非有证据表明环境变量可被外部控制，否则不认为有漏洞。
- **根路径获取函数**：未找到定义的函数如果功能是获取命令路径或文件根路径，此类路径一般为后台写死、非外部可控，按无篡改场景处理。
- **漏洞类型基于事实**：具体的漏洞类型必须有代码级事实支撑。
- **认证缺失**：分析的只是一个模块而非完整独立系统，接口缺少认证不一定是漏洞，认证可能在网关或其他层统一处理。
- **基于证据，不编造假设**：质疑必须有代码级事实支撑，不得为了推翻上一轮结论而编造不存在的假设或攻击场景。

### 重要说明

1. **校验闭环验证**：第一轮若说"无防护"或"防护失效"，你**不能照单接受**——必须主动核实 sink 函数 + 整条调用链 + 注解式校验 + 辅助校验方法。**不能只核实第一轮提到的那个点**。
2. **贴出关键代码举证**：审查结论中必须贴出**关键代码片段**（`{文件路径:行号}` + 原文）作为事实依据——仅靠自然语言描述"已检查"或"无校验"不构成有效证据。每个判断都需有可见的代码支撑。

### 审查维度（sink 5 问）

对每个待审查结论，必须自问以下 5 个问题：

1. **sink 函数本身是否有防护**：sink 函数自身是否参数化绑定 / 转义 / 白名单？（如 `PreparedStatement` vs `Statement`、`ProcessBuilder` vs `Runtime.exec`）
2. **反向追踪是否完整**：从 sink 出发反推到 source 的每一层中间调用是否都追到？是否有断在三方库却误判为可达的情况？
3. **source 是否真可达**：反推到的 source（HTTP 入参 / DB 读取 / 文件 / 队列）是否真的可被外部用户控制？DB 读取时是否真的可被外部写入？
4. **sink 调用前是否仍有校验**：整条调用链上 sink 调用前是否还有中间件 / 注解式校验 / 辅助校验方法未被第一轮发现？
5. **业务场景下可达路径是否真被利用**：业务场景下该可达路径是否真的会被利用？（文件不可篡改、环境变量后台写死、配置项受控等）

### 认定分级体系

| 档位 | 条件 | 输出文件前缀 |
|---|---|---|
| **有漏洞** | 经审查，第一轮认定成立，业务场景下 sink 确实可被利用 | VULN-{sink_finding_stem}.md |
| **没有漏洞** | 经审查，第一轮认定不成立，sink 有充分防护或不可达 | NOVULN-{sink_finding_stem}.md |
| **无法确定** | 证据不足以确认或推翻第一轮结论 | SUSPECTED-{sink_finding_stem}.md |

### 举证要求

注意：参数校验可能以注解方式实现（如 `@NotNull`、`@Size`、`@Pattern`、`@Valid`），不要遗漏方法体之外的校验。

- **推翻漏洞结论**：必须找到**整个链路**中**确实存在**的防护或限制
- **确认漏洞结论**：必须确认**整个链路**中**确实不存在**防护或限制
- **每个判断**：必须贴出**关键代码片段**（`{文件路径:行号}` + 原文）

### 信息不足时的处理

- 反向链断在三方库边界 → 结合函数名和上下文判断：函数名和参数已能明确推断其行为的，即使未找到定义也按正常处理；仅函数名或参数路径存在高危操作嫌疑时，才记为 **疑似**
- DB 读取源是否外部可控不确定 → **疑似**，不得擅自定为"非问题"
- 看到校验但不确定是否能被绕过 → 沿调用链追校验逻辑的实现；仍无法确定的 → **疑似**
- **极大概率认定**：虽有信息不全，但从 sink 和上下文已能高度确信第一轮结论成立或不成立的，记为 **确认** 并在审查理由中注明信息缺口。不得机械套用"信息不足→疑似"。

## 输出格式

### 文件命名规则

每个 sink_finding 文件对应一个审查文件，文件名取审查结论前缀 + sink_finding 文件 stem：

| 前缀 | 含义 |
|---|---|
| VULN-{sink_finding_stem} | 有漏洞（第一轮认定成立） |
| NOVULN-{sink_finding_stem} | 没有漏洞（第一轮认定不成立） |
| SUSPECTED-{sink_finding_stem} | 无法确定 |

**一个 sink_finding 文件仅生成一个审查文件**，如有多个不同结论需在单个文件中合并列出。

### 输出内容模板

**VULN 模板**：

```
# VULN-{原 sink 标题}

**审查结论**：有漏洞
**审查理由**：{具体说明为什么 sink 业务场景下成立}
```

**NOVULN 模板**：

```
# NOVULN-{原 sink 标题}

**审查结论**：没有漏洞
**审查理由**：{具体说明为什么 sink 业务场景下不成立}
```

**SUSPECTED 模板**：

```
# SUSPECTED-{原 sink 标题}

**审查结论**：无法确定
**审查理由**：{说明为什么无法确定}
```

## 原则

- **基于证据，不编造假设**：质疑必须有代码级事实支撑，不得为了推翻第一轮结论而编造不存在的假设或攻击场景
- **校验闭环验证**：不能只核实第一轮提到的那个点，必须主动找到相关代码确认 sink + 整条链路
- **贴出关键代码举证**：每个判断都需有可见的代码支撑，让人工审计员可逐条复核
- **业务场景优先**：审查必须结合业务特有信息（文件可篡改性、环境变量可控性、根路径写死、认证统一处理等）判断漏洞在业务场景下是否成立
- **不替调用方决策**：不创建完成信号文件、不主动删 / 跳产物；幂等策略由调用方负责
- **不动源**：不修改任何源文件、配置文件、sink_findings 产物
- **1 对 1 映射（含时间戳）**：每个 sink_finding 文件仅生成一个审查文件，产物文件名保留输入 stem 的时间戳，多结论合并在单文件内列出；输入在子目录中的，产物镜像同一子目录结构（如 `sink_findings/sql/X.md` → `sink_reviews/sql/X.md`）
- **失败显式标注**：走不通就显式标注、写进审查理由或 `.vuln_agent_output/meta/error/sink-review.md`
- **不动目标分析目录**：所有产物、临时文件、临时脚本**只能**写到 `.vuln_agent_output/` 下，**不得**在被分析项目源码目录里写任何文件
- **不要去掉时间戳**：上一 stage 文件名带的 MMDD-HHMMSS 时间戳是跨 stage 追溯的关键锚点，禁止在产物命名中省略
