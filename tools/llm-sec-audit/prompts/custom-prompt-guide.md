# 自定义分析提示词编写指南

如果预置的四种分析维度（内存安全、敏感信息泄露、注入漏洞、逻辑缺陷）不能满足需求，你可以编写自定义提示词。

## 提示词结构

一个有效的自定义提示词应包含以下部分：

### 1. 角色设定

告诉 LLM 它是什么角色的专家。例如：

```
你是代码安全审计专家，专注于 XXX 问题分析。
```

### 2. 检查项清单

列出需要检查的具体问题类型，每项一行。例如：

```
- XXX：描述
- YYY：描述
- ZZZ：描述
```

检查项越具体，分析结果越精准。避免模糊的描述如"检查安全问题"。

### 3. 输出格式

规定每个发现的问题如何输出，确保结果结构化、可解析。建议格式：

```
### 问题 [编号]

- **类型**：问题所属类型
- **位置**：行号或代码片段
- **严重度**：高 / 中 / 低
- **描述**：问题的具体说明
- **修复建议**：具体的修复方向
```

### 4. 约束要求

- 明确什么该报、什么不该报（减少误报）
- 定义严重度标准
- 说明无问题时的默认输出（如"未发现 XXX 问题"）
- 要求给出具体行号引用

## 示例：自定义"并发安全"分析提示词

```
你是代码安全审计专家，专注于并发安全问题分析。请仔细审查以下代码，找出所有可能导致并发安全问题的隐患。

## 检查项

- **数据竞争**：多个线程/协程同时读写共享变量且无同步保护
- **死锁**：多个锁的获取顺序不一致导致循环等待
- **活锁**：线程反复重试但无法推进
- **锁粒度不当**：锁范围过大影响性能或过小无法保护一致性
- **遗漏锁保护**：共享数据访问缺少必要的锁

## 输出格式

对每个发现的问题，按以下格式输出：

### 问题 [编号]

- **类型**：数据竞争 / 死锁 / 活锁 / 锁粒度不当 / 遗漏锁保护
- **位置**：行号或代码片段
- **严重度**：高 / 中 / 低
- **描述**：问题的具体说明
- **修复建议**：具体的修复方向

## 要求

1. 只报告有实际证据的疑似问题
2. 高=可被实际触发导致数据损坏或死锁；中=需要特定条件；低=理论风险
3. 如果代码中没有发现并发安全问题，输出"未发现并发安全问题"
4. 对每个问题给出具体的行号或代码片段引用
```

## 使用自定义提示词

编写好提示词后，在调用 skill 时选择"自定义"维度，并提供提示词内容。skill 会将其通过 `--prompt` 参数传递给 `llm_prompt_cli`。
