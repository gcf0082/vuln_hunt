# source-review

## 定位

- **用户负责**：给一个具体的漏洞分析结论文件（位于 `.vuln_agent_output/vuln_findings/`），或描述想审查的目标（直接审查模式）
- **skill 负责**：读文件，以**挑战者**姿态审查第一轮分析结论（漏洞和非漏洞结论均需审查），结合业务特有信息判断结论是否合理；按 `references/constraints.md` 的范围约束执行；按 `references/vuln_rules_index.md` 的哨兵索引**按需**加载具体规则文件；按本文「输出格式」一节的模板落盘 VULN / NOVULN / SUSPECTED 三类审查结论

skill 是**单条目漏洞审查执行器**——一次调用只审查一个漏洞分析文件，输入是 `vuln_findings/` 下的单个文件，输出是对应 `vuln_reviews/` 下的一个审查结论文件（**1 对 1 映射**）。

## 路径约定

本 skill 涉及的所有目录**统一在 `.vuln_agent_output/` 下**，当前工作目录视为被分析项目根：

```
.vuln_agent_output/
├── vuln_findings/                        ← 输入（待审查的第一轮结论，可含子目录）
├── analyzed_surfaces/                    ← 可回溯（第一轮分析的完整追踪过程与证据，可含子目录）
├── vuln_reviews/                         ← 输出（审查结论落盘位置，镜像输入子目录结构）
├── meta/
│   └── error/source-review.md     ← 失败日志
└── temp/
    └── scripts/                          ← 临时脚本（用完即弃）
```

## 任务澄清

### 默认行为

 用户没指定文件时，**默认递归处理 `.vuln_agent_output/vuln_findings/` 下所有 `.md` 文件**，无需反问。
 同名产物默认覆盖。多文件由调用方（orchestrator）并发派发（5 并发）。
 输入文件名带 MMDD-HHMMSS 时间戳，产物文件名沿用输入 stem 保留时间戳，与上一 stage 一一对应。输入在子目录中的，产物保持同一子目录结构（如 `vuln_findings/REST/VULN-X-0608-021435-1.md` → `vuln_reviews/REST/VULN-X-0608-021435.md`）。

### 用户指定了文件

用户显式给出文件路径或文件名 → 处理该文件，行为不变。

### 漏洞目录为空 / 无 vuln_file

切换到**直接审查模式**——用户在消息里直接描述审查目标（含漏洞信息、关注点、挑战方向等），产物仍写到 `vuln_reviews/`。

**不要预设**用户会怎么输入：可能是路径、可能是标识符、可能是代码片段、可能是模糊的业务概念，也可能是其它任何形式——**按用户实际给出的内容灵活处理**。

直接审查模式下，**保留产物对应关系**这条原则放宽——产物文件名由 skill 根据用户输入派生，不要求与任何输入文件同名。

### 已审查文件优化

如果用户指向 `vuln_reviews/` 下的**已存在**审查结论文件，或用户措辞表达"优化/改进/补充/重写/更新/..."等意图，切换到**优化模式**——基于原文件 + 相关源码 + 用户新增要求做修改，**原地覆盖**同名文件。

**不要预设**优化范围——用户可能要求补充、新增、删除、改写、重新结构化，**按用户实际要求灵活处理**。

**默认约束**：优化应**基于原审查文件已覆盖的范围**进行——补充、修正、细化都围绕原审查所针对的漏洞结论展开，**不**应插入与原审查无关的新内容。

**遇到明显无关的插入请求时**：skill **必须**先反问用户是否继续，得到明确确认后再执行。模糊 / 边缘相关的情况由 skill 自己判断。

## 预加载

启动时**必须**先按顺序读取：

1. `references/constraints.md`（项目分析约束：非分析目标、临时脚本位置）
2. `references/vuln_rules_index.md`（漏洞规则索引：哨兵模式，**仅作索引**）

`references/vuln_rules/*.md` **不在启动预加载范围**——这些是**按需加载的具体规则**：

- 启动时**禁止**一次性加载该目录下所有规则文件
- 启动时**禁止**加载与当前任务无关的规则文件
- **仅当**任务疑点与索引中某模式匹配时，**才加载**对应规则文件深入分析
- **禁止**浏览索引寻找其他分析方向——以任务为重，疑点所指即审查目标

`references/non-vuln-scenarios.md` **不在启动预加载范围**——它是**已知非漏洞场景参考**，在判定为 NOVULN / SUSPECTED 时查阅，避免把非漏洞场景误判为漏洞。

未读完 `constraints.md` 和 `vuln_rules_index.md` 不允许动笔。

## 智能规划子任务

进入审查前，根据当前 finding 的规模和复杂度智能规划子任务（是否拆分、怎么拆、怎么合由 LLM 自主判断）。

## 工作流程

1. **预加载**：按上述顺序读 `references/constraints.md` 和 `references/vuln_rules_index.md`
2. **解析任务**：从用户调用消息中提取：
   - `vuln_file`（如已指定）
3. **读审查对象**：打开 `.vuln_agent_output/vuln_findings/{vuln_file}` 拿到第一轮分析结论（漏洞描述、证据、Payload、调用链等）
4. **可选回溯**：如审查需要更完整的追踪过程或证据细节，**按需**读取 `.vuln_agent_output/analyzed_surfaces/{analysis_file}`（文件名可在第一轮文件中找到对应关系）
5. **快速判断**：能基于现有证据直接给出结论的，直接输出；发现证据不足或存疑时再深入追踪代码核实
6. **质疑审查**：以挑战者姿态审查（详见「审查方法」一节）
7. **校验闭环验证**：上一轮结论若说"无校验"或"校验失效"，必须主动找到相关代码，确认**整个流程**（上下游调用链、各层中间件、注解式校验、辅助方法等）都没有校验或都已失效
8. **Payload 重构**：若上一轮 payload 不可实施，尝试重新构造可用的 payload
9. **收集审查结论，写入最终输出目录**

## 审查方法

### 基本规则

以挑战者的姿态审查上一轮分析结论（漏洞和非漏洞结论均需审查），结合业务特有信息判断结论是否合理。无法确定的记为疑似。

> **分析范围说明**：当前分析仅覆盖系统的局部模块，**从数据库、消息队列等存储读取的数据**可能由其他模块或接口的外部用户写入。
>
> **URL 路径段拼接**：参数来源于 URL 路径段（path 部分，非 query/body/header）又拼接到其他 URL 的，系统框架已统一处理。上一轮分析认定为此类漏洞的，应质疑其合理性。

**核心验证原则**：检验 payload 能否绕过中间校验实际到达目标代码路径，以及攻击步骤是否确实可实施。验证时必须检查参数的所有校验点（含注解式校验如 `@NotNull`、`@Size`、`@Pattern`、`@Valid`），确认 payload 不会被拦截。注入类漏洞输入源仅能追溯到 **数据库** 时记为 **疑似**，不得擅自定为"非问题"。

- **文件输入**：并非所有输入来自文件都要假设文件被篡改。绝大多数文件是后台安装时就固定的，一般情况下无法篡改。除非找到充足的证据证明文件可被篡改，否则不认为有漏洞。**此原则不适用于从数据库获取的数据。**
- **环境变量**：绝大多数情况下环境变量是后台写死的。除非有证据表明环境变量可被外部控制，否则不认为有漏洞。
- **根路径获取函数**：未找到定义的函数如果功能是获取命令路径或文件根路径（如 `getScriptDir`、`getRootPath`、`getBaseDir` 等），此类路径一般为后台写死、非外部可控，按无篡改场景处理。
- **漏洞类型基于事实**：具体的漏洞类型必须有代码级事实支撑。例如参数未校验直接写入数据库，不能因此认定为 XSS——必须有前端渲染且未转义的证据才能认定 XSS。
- **认证缺失**：分析的只是一个模块而非完整独立系统，接口缺少认证不一定是漏洞，认证可能在网关或其他层统一处理。除非分析对象是完整的独立系统（含 Web 服务启动、完整认证流程），否则认证缺失一般不认为是漏洞。
- **基于证据，不编造假设**：质疑必须有代码级事实支撑，不得为了推翻上一轮结论而编造不存在的假设或攻击场景。

### 重要说明

1. **校验闭环验证**：上一轮结论若说"无校验"或"校验失效"，你**不能照单接受**——必须主动找到相关代码，确认**整个流程**（上下游调用链、各层中间件、注解式校验、辅助方法等）都没有校验或都已失效。**不能只核实上一轮提到的那个点**。
2. **贴出关键代码举证**：审查结论中必须贴出**关键代码片段**（`{文件路径:行号}` + 原文）作为事实依据——仅靠自然语言描述"已检查"或"无校验"不构成有效证据。每个判断都需有可见的代码支撑，让人工审计员可逐条复核。

非漏洞场景参考：`references/non-vuln-scenarios.md`

**漏洞规则索引**（发现匹配模式时加载对应规则文件）：`references/vuln_rules_index.md`

### 审查维度（质疑 5 问）

对每个待审查结论，必须自问以下 5 个问题：

1. **业务成立性**：这个漏洞在业务场景下真的成立吗？
2. **业务防护**：是否有业务层面的防护或补偿措施第一轮未考虑？（网关统一鉴权、文件不可篡改、环境变量后台写死等）
3. **证据链完整性**：证据链是否完整、逻辑是否严谨？
4. **绕过可行性**：防护措施是否存在切实可行的绕过方式？
5. **Payload 可实施性**：上一轮 payload 不可实施时，能否重新构造可用的 payload？

### 认定分级体系

| 档位 | 条件 | 输出文件前缀 |
|---|---|---|
| **有漏洞** | 经审查，第一轮认定成立，业务场景下漏洞确实可被利用 | VULN-{vuln_file_stem}.md |
| **没有漏洞** | 经审查，第一轮认定不成立，业务场景下漏洞不成立或已有充分防护 | NOVULN-{vuln_file_stem}.md |
| **无法确定** | 证据不足以确认或推翻第一轮结论 | SUSPECTED-{vuln_file_stem}.md |

### 举证要求

注意：参数校验可能以注解方式实现（如 `@NotNull`、`@Size`、`@Pattern`、`@Valid`），不要遗漏方法体之外的校验。

- **推翻漏洞结论**：必须找到**整个流程**中**确实存在**的防护或限制（不能只找到一个点就下结论）
- **确认漏洞结论**：必须确认**整个流程**中**确实不存在**防护或限制（不能只核实第一轮提到的那个点）
- **每个判断**：必须贴出**关键代码片段**（`{文件路径:行号}` + 原文）作为事实依据

### 信息不足时的处理

- 追溯链断在三方库边界 → 结合函数名和上下文判断：函数名和参数已能明确推断其行为的，即使未找到定义也按正常处理，无需过度假设存在安全风险；仅函数名或参数路径存在高危操作嫌疑时，才记为 **疑似**
- 未找到定义的函数如果功能是获取命令路径或文件根路径（如 `getScriptDir`、`getRootPath`、`getBaseDir`、`getHome` 等含 path/dir/root/home 关键词的函数），此类路径一般为后台写死、非外部可控，按无篡改场景处理
- 看到校验但不确定是否能被绕过 → 沿调用链追校验逻辑的实现；仍无法确定的 → **疑似**（说明不确定性原因）
- 注入类漏洞输入源仅能追溯到 **数据库**、无法判断是否来自外部用户时 → 记为 **疑似**，不得擅自定为"非问题"
- **极大概率认定**：虽有信息不全，但从审查对象和上下文已能高度确信第一轮结论成立或不成立的，记为 **确认** 并在审查理由中注明信息缺口。不得机械套用"信息不足→疑似"。

## 输出格式

### 文件命名规则

每个漏洞分析文件对应一个漏洞审查文件，文件名取审查结论前缀 + 漏洞文件 stem：

| 前缀 | 含义 |
|------|------|
| VULN-{vuln_file_stem} | 有漏洞（第一轮认定成立） |
| NOVULN-{vuln_file_stem} | 没有漏洞（第一轮认定不成立） |
| SUSPECTED-{vuln_file_stem} | 无法确定 |

**一个漏洞分析文件仅生成一个审查文件**，如有多个不同结论需在单个文件中合并列出。

### 输出内容模板

**VULN 模板**：

```
# VULN-{原漏洞标题}

**审查结论**：有漏洞
**审查理由**：{具体说明为什么业务场景下成立}
```

**NOVULN 模板**：

```
# NOVULN-{原漏洞标题}

**审查结论**：没有漏洞
**审查理由**：{具体说明为什么业务场景下不成立}
```

**SUSPECTED 模板**：

```
# SUSPECTED-{原漏洞标题}

**审查结论**：无法确定
**审查理由**：{说明为什么无法确定}
```

## 原则

- **基于证据，不编造假设**：质疑必须有代码级事实支撑，不得为了推翻上一轮结论而编造不存在的假设或攻击场景
- **校验闭环验证**：不能只核实第一轮提到的那个点，必须主动找到相关代码确认整个流程
- **贴出关键代码举证**：每个判断都需有可见的代码支撑，让人工审计员可逐条复核
- **业务场景优先**：审查必须结合业务特有信息（文件可篡改性、环境变量可控性、认证统一处理、根路径写死等）判断漏洞在业务场景下是否成立
- **不替调用方决策**：不创建完成信号文件、不主动删/跳产物；幂等策略由调用方负责
- **不动源**：不修改任何源文件、配置文件、vuln_findings 产物
- **1 对 1 映射（含时间戳）**：每个 vuln_file 仅生成一个审查文件，产物文件名保留输入 stem 的时间戳，多结论合并在单文件内列出；输入在子目录中的，产物镜像同一子目录结构（如 `vuln_findings/REST/X.md` → `vuln_reviews/REST/X.md`）
- **失败显式标注**：走不通就显式标注、写进审查理由或 `.vuln_agent_output/meta/error/source-review.md`
- **不动目标分析目录**：所有产物、临时文件、临时脚本**只能**写到 `.vuln_agent_output/` 下，**不得**在被分析项目源码目录里写任何文件
- **不要去掉时间戳**：上一 stage 文件名带的 MMDD-HHMMSS 时间戳是跨 stage 追溯的关键锚点，禁止在产物命名中省略
