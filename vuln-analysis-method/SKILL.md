# vuln-analysis-method

## 核心框架

对每个危险操作执行三问：

```
① 能造成什么危害？（识别危险语义）
② 参数谁可控？（追溯数据来源）
③ 到这一步前有有效防护吗？（验证防护边界）
```

## 技术手段

- **数据流跟踪**：追踪数据从源头到危险操作的传递和变换，回答 ②
- **调用链分析**：追踪函数间调用关系，构建入口到 sink 的完整路径，回答 ① 和 ③

## 方向自适配

| 任务特征 | 方向 |
|---|---|
| 已知入口 | 自上而下：从入口出发，数据流跟踪逐跳前进 |
| 已知漏洞类型 | 自下而上：从 sink 出发，调用链反向回溯 |
| 全面审计 / 不明确 | 交汇验证：双向同时推进，在中间汇合确认 |

分析过程中可根据发现动态切换方向。

## 纪律

- **不跳步**：每跳走一遍三问
- **不假设**：以源码为证据，不凭经验编造
- **不轻信**：防护需验证能否绕过，而非只看有没有
- **不中止**：未定义函数结合上下文（函数名/参数/包名/使用方式）给出推测，标注不确定性后继续推进
- **不沉默**：未知不是安全，显式标注证据等级

## 举证等级

| 等级 | 适用场景 |
|---|---|
| 已确认 | 读到源码，行号可定位 |
| 已推断 | 函数未定义但上下文充分，可合理推测行为 |
| 需人工确认 | 关键证据缺失，或依赖较多假设 |
