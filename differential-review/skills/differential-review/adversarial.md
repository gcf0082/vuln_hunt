# 对抗性漏洞分析（Phase 5）

通过攻击者建模发现漏洞的结构化方法论。

**使用时机：** 完成深度上下文分析（Phase 4）后，对所有 HIGH RISK 变更应用。

---

## 1. 定义攻击者模型

**WHO——攻击者是谁？**
- 未认证的外部用户
- 已认证的普通用户
- 恶意管理员
- 被攻陷的上游服务
- 内部网络访问者

**WHAT——他们拥有什么访问权限？**
- 公开 API 端点
- 已认证用户会话
- 特定的角色/权限
- API 令牌或 JWT

**WHERE——他们在哪里与系统交互？**
- HTTP API 端点
- WebSocket 连接
- 消息队列
- 外部 API 回调

---

## 2. 识别具体攻击向量

```
入口点: [攻击者可访问的具体函数/端点]

攻击序列:
1. [具体的 API 调用及其参数]
2. [如何到达脆弱代码]
3. [脆弱代码中发生什么]
4. [达到的影响]

可访问性证明:
- 显示函数是公开的
- 演示攻击者拥有所需权限
- 证明攻击路径通过实际接口存在
```

---

## 3. 评估可利用性

**简单：** 通过公开 API 利用，无需特殊权限
- 单次请求/调用
- 普通用户访问级别
- 无需复杂条件

**中等：** 需要特定条件或提升的权限
- 多个步骤或时序要求
- 可获取的提升权限
- 特定系统状态

**困难：** 需要特权访问或罕见条件
- 管理员/超级用户权限
- 罕见边缘情况
- 需要大量资源

---

## 4. 构建完整利用场景

```
攻击者初始状态:
[攻击者开始时拥有什么]

逐步利用:
步骤 1: [通过可访问接口的具体操作]
  - 请求: [确切 HTTP 请求/调用]
  - 参数: [具体值]
  - 预期结果: [发生什么]

步骤 2: [下一步操作]
  - 请求: [确切 HTTP 请求/调用]
  - 为何成功: [引用代码变更]
  - 系统状态变化: [什么改变了]

步骤 3: [最终影响]
  - 结果: [实现的具体危害]
  - 证据: [如何验证影响]

具体影响:
[具体的、可衡量的影响——不是"可能造成问题"]
- 泄露的确切数据类型和数量
- 提升的特定权限级别
- 被操纵的金融/业务数据
```

## 5. 与基线上下文交叉引用

从基线分析（参见 [methodology.md](methodology.md#pre-analysis-baseline-context-building)）检查：
- 这是否违反了系统级不变量？
- 这是否打破了信任边界？
- 这是否绕过了验证模式？
- 这是否是之前修复的回归？

## 示例：完整对抗分析

**变更：** 从 `DELETE /api/users/{id}` 中移除了授权检查

### 1. 攻击者模型
- **WHO:** 已认证的普通用户（角色: `USER`）
- **ACCESS:** 有效的 JWT 令牌，可调用 `/api/**` 端点
- **INTERFACE:** `DELETE /api/users/{id}` 端点

### 2. 攻击向量
**入口点:** `DELETE /api/users/42`

**攻击序列:**
1. 攻击者向 `DELETE /api/users/42` 发送带有自己 JWT 的请求
2. 代码不再检查当前用户是否为 `admin` 或是否拥有 `id=42` 的账户
3. 用户 ID 为 42 的账户被删除
4. 系统数据丢失

**可访问性证明:** 端点需要 JWT（任何已认证用户），无角色检查

### 3. 可利用性
**评级:** 简单
- 单次 HTTP 请求
- 需要有效 JWT（已认证用户）
- 无需特殊状态

### 4. 利用场景
**攻击者状态:** 拥有有效 JWT 的普通用户

**利用:**
```python
# 步骤 1: 发送不带 role 检查的 DELETE 请求
import requests
headers = {"Authorization": "Bearer user_jwt_token"}
resp = requests.delete("https://api.example.com/users/42", headers=headers)

# 步骤 2: 服务器接受请求（授权检查被移除）
# 步骤 3: 用户 42 从数据库中被删除

# 影响: 任意用户被删除，数据永久丢失
```

**影响:**
- 任何已认证用户可删除任何其他用户账户
- 影响 N 个用户账户
- 业务数据完整性被破坏

### 5. 基线违反
- 违反不变量："只有用户本人或管理员可以删除账户"
- 破坏信任边界：将管理操作降级为普通用户操作
- 回归：角色检查在 commit xyz789 中被添加（"按审计结果添加角色检查"）
